# 055-03 PLAN - Package Hygiene And Transport Privacy

## 🎯 Objective

Separate chain-state unlinkability from transport privacy by defining one package privacy policy and one node-owned boundary for future transport anonymity work.

## 🔒 Truth Boundary

- State unlinkability does not imply transport anonymity.
- `TxPackage`, forwarding bundles, logs, and export artifacts remain sensitive material.
- `crates/z00z_networks/onionnet` is the reserved namespace for future transport-anonymity work.
- No protocol-enforced anonymity overlay is shipped today.

## 🔗 Cross-Spec Dependencies

- `10-Privacy-Anonymity.md` backlog rows `PRIV-055-03` and `PRIV-055-05`
- `050-Offline-Tx-Spec.md` for portable package sensitivity and routing boundaries
- `028-01-PLAN.md` and `031-08-PLAN.md` for truthful proof-payload and checkpoint-artifact wording

## 📌 Numbered Tasks

1. Define one shared privacy policy for tx packages, forwarding bundles, exports, backups, relay handoff, and log surfaces.
2. Mark every path where package bytes leave wallet-local trust and require encryption or explicit redaction on those paths.
3. Specify verification rules that fail if plaintext receiver artifacts or package payloads are logged or exported through protected surfaces.
4. Keep future transport-anonymity implementation work under the OnionNet crate boundary instead of leaking overlay logic into wallet ownership modules.
5. Separate deployment-time privacy helpers from protocol-level anonymity claims in docs, tests, and future implementation gates.

## ✅ Exit Criteria

- Package hygiene is described by one explicit policy instead of scattered assumptions.
- OnionNet remains the only approved namespace for future transport overlay work.
- Documentation no longer conflates package-local verification with transport anonymity.
