# 055-01 PLAN - Field-Native Pack Migration

## 🎯 Objective

Define the migration path from the live wallet `ZkPack_v1` contract to a future field-native pack contract without overstating current OWF or proof parity.

## 🔒 Truth Boundary

- `ZkPack_v1` using `ChaCha20Poly1305` is the live truth today.
- The repository does not yet have field-native pack parity at the wallet or proof boundary.
- Any migration must preserve request-bound receive behavior and current wallet decrypt semantics until parity is proven.

## 🔗 Cross-Spec Dependencies

- `10-Privacy-Anonymity.md` backlog row `PRIV-055-01`
- `040-Spend-Proof-Spec.md` for public proof-boundary limits
- `050-Offline-Tx-Spec.md` for portable package sensitivity and offline package handling
- `028-01-PLAN.md` and `031-08-PLAN.md` for truthful checkpoint and proof-payload semantics

## 📌 Numbered Tasks

1. Freeze the current `ZkPack_v1` contract, fixture corpus, associated-data bindings, and privacy wording as the migration baseline.
2. Define one crypto-owned field-native pack contract that can coexist with the current wallet pack without changing the live portable package truth.
3. Specify dual-read or equivalent compatibility gates so historical wallet outputs remain decryptable during migration.
4. Define OWF-parity acceptance rules that compare decrypt, commitment-open, and associated-data behavior across current and future pack variants.
5. Require rollback-safe wording: the old pack remains current truth until wallet, tests, and proof-facing consumers all accept the new contract.

## ✅ Exit Criteria

- Current-pack truth is frozen and cannot drift silently.
- Future field-native pack work has one explicit ownership boundary.
- Migration wording does not claim completed OWF or proof parity before tests prove it.
