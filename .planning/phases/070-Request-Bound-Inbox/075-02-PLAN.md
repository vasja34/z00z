# 055-02 PLAN - Request-Bound Inbox Helper

## 🎯 Objective

Add a helper-only inbox or routing layer that reduces scan cost while preserving the request-bound privacy lane and keeping compatibility paths explicitly secondary.

## 🔒 Truth Boundary

- Request-bound receive is the preferred privacy lane.
- Card-only and plain receive remain compatibility-only.
- No shipped inbox helper exists today.
- `tag16` is a prefilter only and must not become an ownership or identity oracle.

## 🔗 Cross-Spec Dependencies

- `10-Privacy-Anonymity.md` backlog rows `PRIV-055-02`, `PRIV-055-04`, and `PRIV-055-06`
- `050-Offline-Tx-Spec.md` for routing-artifact and package-sensitivity rules
- `040-Spend-Proof-Spec.md` for admission-boundary versus public-proof boundary discipline

## 📌 Numbered Tasks

1. Define a helper-record schema derived only from request-bound context, per-request route buckets, and encrypted hints.
2. Forbid persistence of `owner_handle`, `view_pk`, or any stable receiver identifier in helper records.
3. Add compatibility-boundary rules so raw-card or plain receive cannot be described or tested as privacy-equivalent to the request-bound lane.
4. Define request rotation requirements so helper lookups change when the request surface changes.
5. Add wallet-local collision, backoff, and deferred-processing policy for `tag16` or helper-prefilter overload.
6. Define negative verification rules proving helper collisions do not become an ownership oracle and do not force blocking receive behavior.

## ✅ Exit Criteria

- Inbox-helper planning stays helper-only and off-consensus.
- Compatibility receive paths remain explicitly subordinate.
- Abuse-handling rules exist for collision-heavy scan conditions.
