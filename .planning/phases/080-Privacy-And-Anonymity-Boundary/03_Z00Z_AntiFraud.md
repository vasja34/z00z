# Введение

Этот документ описывает **Anti-Fraud архитектуру Z00Z** — набор протокольных правил и криптографических механизмов, которые позволяют сочетать приватность честных пользователей с жёсткой и предсказуемой реакцией сети на мошенничество. В центре подхода — лёгкая схема **`link_tag`** (криптографический «штамп личности») и ончейн-процедура **штрафа через DAO** с автоматической разблокировкой после оплаты.

### 🎯 Цели

* Сохранить **unlinkability** и анонимность в штатном режиме.
* Обеспечить **обнаружение и блокировку** нарушителя после доказанного фрода без участия централизованного арбитра.
* Сделать проверку валидаторами **дешёвой и детерминированной** (без тяжёлых zk-доказательств).
* Задать  **прозрачную экономику наказаний** : штраф =  *2× ущерб* , поступающий в DAO.

### 🧩 Ключевые идеи

* **`link_tag = P_in^a` + `link_proof` (Schnorr)** : при каждой трате публикуется метка и доказательство знания корневого секрета `a` без его раскрытия; метка меняется от Coin к Coin, но «несёт» скрытый отпечаток личности.
* **DAO-чёрный список** : после доказанного фрода протокол добавляет `a_bad` (или эквивалент) в on-chain список; валидаторы сравнивают присланные `link_tag` с вычисленным `P_in^{a_bad}` и **отклоняют** такие SpendTx.
* **`PenaltyPaymentTx`** : единственная разрешённая tx от заблокированного участника — оплата штрафа в DAO  **ровно на сумму 2× fraud_amount** ; после включения запись меняет статус на *cleared* и блокировка автоматически снимается.

### 🛡️ Модель угроз (что мы предотвращаем)

* **Double-spend / replay** одних и тех же входов.
* **Identity rotation** : попытку обойти санкции простой сменой ключей.
* **Ack/commitment-мошенничество** в подтверждениях протокольных обещаний.
* **Spam-атаки транзакциями** от уже заблокированного участника (до оплаты штрафа).

### 🧭 Дизайн-принципы

* **Privacy-by-default** : нет глобальных идентификаторов; метки одноразовые и не связываемы между собой без знания `a`.
* **Revocable anonymity** : при доказанном фроде протоколно включается «режим ответственности» через DAO-чёрный список.
* **Light-verify** : верификация сводится к хешам и двум линейным равенствам уровня Schnorr (без zk-SNARK).
* **On-chain исполнение санкций** : правила наказаний и разблокировок детерминированы и публичны.

### 🏗️ Компоненты и артефакты

* **SpendTx.inputs** : содержат `pubkey_in`, `sig_eph`, `link_tag`, `link_proof`, `nullifier`, `inclusion_proof`.
* **FraudProofTx** : публикует доказательства конфликта/фрода и (при необходимости) раскрывает `a_bad`.
* **DAO Blacklist** : ончейн структура с полями `identity_tag`, `penalty_due = 2× fraud_amount`, `status`.
* **PenaltyPaymentTx** : специальная транзакция оплаты штрафа на DAO-адрес, проверяет связь с заблокированной личностью через тот же `link_tag`.

### ✅ Гарантии и компромиссы

* **Гарантии** : неизбежная детекция повтора/фрода; невозможность обойти бан сменой адреса; автоматическая разблокировка после оплаты.
* **Компромиссы** : наличие управляемой трассируемости *ex post* (после фрода) — осознанный отказ от «абсолютной» неотзывной анонимности в пользу безопасности сети и возмещения ущерба.

# 1. Интуиция и формулы

## 🔗 Что такое `link_tag` и зачем он нужен

| Вопрос                             | Краткий ответ                                                                                                                                                                                                                                                                      |
| ---------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Что это?**                 | Точка элл. кривой `L = P_in^a`, вычисляемая из адреса входной монеты `P_in = g^x_in` и **корневого** секрета владельца `a`.                                                                            |
| **Зачем?**                    | Дать сети мягкий механизм “связать” все траты одной личности**только тогда, когда это понадобится** (например, для блок‑листа мошенника) **и не раньше**. |
| **Как проверяется?** | Каждый `link_tag` сопровождается доказательством (`link_proof`)‑Schnorr: валидатор убеждается, что тот же отправитель знает `a`, но само `a` не публикуется.                      |
| **Где хранится?**       | Внутри**каждого входа** `SpendTx` — под полем `link_tag` + `link_proof`.                                                                                                                                                                             |

---

### 1. Как формируется `link_tag`

```
# входная монета
P_in = g ^ x_in          # публичный ключ монеты
a    = secret_identity   # постоянный корневой скаляр кошелька

link_tag L = P_in ^ a    # ≡ g^(x_in · a)

```

- Для **каждой** тратимой монеты получается свой L (потому что P_in разные),

  но все они «несут отпечаток» одного и того же `a`.

---

### 2. Как доказывается корректность `link_tag`

> Валидатору нужно знать, что автор действительно знает a, иначе можно подделать L.

### Шаги Alice (отправителя)

```

# 🔄 1. Генерируем свежий случайный скаляр
v ←$ Fq                 # 🔄 v — равномерно из поля Fq

# 📍 2. «Коммитмент» к v на основе адреса монеты
T = P_in ^ v            # 📍 временная точка (как R в классическом Schnorr)

# 🏷️ 3. Формируем challenge через Fiat–Shamir
c = Hash("Z00Z_SPEND" ‖ P_in ‖ L ‖ T ‖ tx_id)
                      # 🏷️ domain_tag + все критичные поля → никакой ре‑плей

# 🖋️ 4. Ответ с участием корневого секрета a
r = v + a · c   mod q  # 🖋️ r = v + a·c (mod q)

# 📦 5. Готовый доказательный пакет
link_proof = (c, r)    # 📦 отправляется в поле link_proof входа

```

### Проверка узлом

```
P_in^r  ?=  T · L^c
```

***Если равенство выполняется — владелец знает `a`, а L не подделан.***

---

### 3. Где `link_tag` находится в SpendTx

```yaml
inputs:
  - context_id: 0x5aa9…
    commitment: "C_in_1"
    pubkey_in: 0x02…
    sig_eph: { c: … , r: … }

    link_tag: 0x03…                # ← ИМЕННО ЗДЕСЬ
    link_proof: { c: … , r: … }    # подтверждение L = P_in^a

    nullifier: 0x11e4…
    inclusion_proof: <…>

```

Каждый вход содержит **три** взаимосвязанных элемента:

1. `sig_eph` — доказывает владение *x_in*.
2. `link_tag` + `link_proof` — доказывают правильность `L = P_in^a`.
3. `nullifier` — предотвращает двойную трату (зависит от `a`).

---

### 4. Как `link_tag` помогает при black-list

- Если когда‑то раскрывается «плохой» `a_bad`, валидаторы могут заранее вычислять

  `L_bad = P_any ^ a_bad` для **любой** входящей монеты и моментально отклонять такие SpendTx.
- Честного пользователя это не заденет, потому что его `a` другое.

---

### 📝 Итог

- **`link_tag`** — криптографический “штамп” личности, вычисляемый **при каждой** трате, но не раскрывающий сам корневой ключ.
- **Проверяется** простым Schnorr‑равенством, поэтому дешёв для валидаторов.
- Даёт сети возможность **отключить** злоумышленника единым правилом, не ломая приватность честных участников.

# 🔗Advanced Explanation `link_tag` — «криптографическая подпись личности»

Ниже на реальных числовых примерах (упрощённые скаляры) и с подробными комментариями показано, **почему** `link_tag` не подделывается, как он «приклеивается» к хозяину, к полной транзакции и к подписи, а также — как позволяет сети наказывать злоумышленника.

### 0. Стартовые данные (игрушечные числа)

| Параметр                      | Значение              | Комментарий                                                          |
| ------------------------------------- | ----------------------------- | ------------------------------------------------------------------------------- |
| Порядок группы `q`     | **97**                  | маленькое простое, чтобы видеть арифметику |
| Базовая точка `g`       | **5**                   | вместо EC‑точки берём генератор ℤ₉₇               |
| ***Личность Alice***  |                               |                                                                                 |
| `a` (корневой секрет) | **17**                  | хранится только в кошельке                               |
| ***Монета***            |                               |                                                                                 |
| `x_in` (адрес‑скаляр)  | **13**                  | монета принадлежит Alice                                       |
| `P_in = g^x_in mod q`              | 5¹³ mod 97 =**11** | публичный адрес монеты                                      |

*(В реальной кривой это точки, но математика линейная так же.)*

---

### 1. Криптографическая **неизменяемость** `link_tag`

```
# Рассчитанный link_tag
L = P_in^a mod q = 11¹⁷ mod 97 = 22

```

*Чтобы кто‑то подделал `L`, ему нужно найти `L' ≠ 22` и **одновременно** предъявить
пару чисел `(c', r')`, проходящих проверку:*

```
11^r' ≡ T · (L')^c'  (mod 97)

```

*Но T зависит от случайного `v`, а `c' = Hash(tag‖…‖T)` формируется Fiat–Shamir’ом:
подделать всё в комплексе без знания `a` — статистически невозможно (аналог Schnorr‑sign).*

---

### 2. **Однозначная привязка** к владельцу

*Для другой личности (`a₂ = 23`) `link_tag` будет:*

```
L₂ = 11²³ mod 97 = 63   # совершенно иное
```

*Блок‑листу достаточно хранить один «плохой» `a_bad`;
любой `P_any` → `L_bad = P_any^{a_bad}` при проверке сразу виден.*

---

### 3. **Привязка к транзакции** (включён `tx_id`)

```
c = Hash("Z00Z_SPEND" ‖ P_in ‖ L ‖ T ‖ tx_id)

```

*Если Alice хотя бы изменит список выходов ⇒ меняется `tx_id` ⇒
меняется `c` ⇒ равенство проверки ломается ⇒ валидатор отклонит tx.
То есть `link_tag` валиден **только** в контексте данного полного тела SpendTx.*

---

### 4. **Связь с подписью** `sig_eph`

`*sig_eph` доказывает, что Alice знает `x_in`;`link_proof` — что она знает тот же `a`.
Вместе это означает:*

```
(знаю x_in)  ∧  (знаю a)  ⇒  владею монетой P_in^x_in  и  «личностью» a

```

*Таким образом валидатор убеждается, что автор — именно действующий владелец и никто иной.*

---

### 5. 🛡️ Slashing / бан через `link_tag`

1. **Факт мошенничества** (double‑spend, поддельный ACK и т.п.) публикуется в `FraudProofTx`, **где раскрывается *корневой* секрет `a_bad` ил исам `link_tag_bad`**.
2. Узлы добавляют `a_bad` в **чёрный список**.
3. При каждой новой трате они вычисляют:

   ```
   L_test = P_in^{a_bad}
   if L_test == L_in_tx  ⇒  reject + slash_deposit

   ```
4. Все будущие транзакции злоумышленника автоматически отклоняются, а его залог (deposit) сгорает или уходит пострадавшей стороне.

---

### 🧩 Численный пример «бан»

*Пусть валидаторы занесли `a_bad = 17` (Alice замечена в фроде).
Любая новая монета `P_new` проверяется:*

```
L_test = P_new^17  (mod 97)
# сравниваем с L, присланным в SpendTx

```

***Если совпало — транзакция блокируется.* сеть блокирует все будущие транзакции Alice (узнанные по `link_tag`) до тех пор, пока она не внесёт штраф в фонд DAO, равный удвоенной сумме выявленного мошенничества.**

---

### ✅ Итого

| Свойство                                 | Как обеспечивает `link_tag`                                                                                                                                                                                                                                                            |
| ------------------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Неизменяемость**           | Сшит со случайным `v` и Fiat–Shamir‑хэшем ⇒ подделка без `a` невозможна.                                                                                                                                                                               |
| **Привязка к владельцу** | Одно и то же `a` даёт уникальный «отпечаток» во всех его тратах.                                                                                                                                                                                     |
| **К транзакции**                | В хэш‑вызове присутствует `tx_id`; любое изменение тела делает proof недействительным.                                                                                                                                                |
| **К подписи**                      | Проверка требует знания**и** `x_in`, **и** `a` — оба секрета одного лица.                                                                                                                                                                 |
| **Slashing**                               | Раскрытый `a_bad` превращает все будущие `link_tag` злоумышленника в детектор фрода, позволяя валидатору мгновенно банить такие SpendTx и переводить его депозит в штраф. |

**`link_tag` — лёгкая, но мощная «нить», которая соединяет монеты одной личности ровно тогда, когда это действительно нужно для защиты протокола.**

# 🚨 Как работает **штраф DAO = FraudAmount × 2** вместо «сожжённого депозита»

> Ниже — подробная логика, как сеть блокирует все будущие транзакции Alice (узнанные по link_tag) до тех пор, пока она не внесёт штраф в фонд DAO, равный удвоенной сумме выявленного мошенничества.

---

### 1. Поймали фрод → `FraudProofTx`

| Поле         | Значение                                                                               |
| ---------------- | ---------------------------------------------------------------------------------------------- |
| `a_bad`        | раскрытый корневой секрет Alice**или** сам `link_tag_bad` |
| `fraud_amount` | суммарная номинальная стоимость двойной траты         |
| `evidence`     | подпись Alice (`sig_ack`/`sig_eph`) + конфликтующие SpendTx          |

**✅ Проверка валидатором:** убедиться, что два SpendTx делят один и тот же `nullifier`

или нарушают ack‑обещание.

---

### 2. Добавляем **запись блокировки** в DAO‑контракт

```yaml
BlacklistEntry:
  identity_tag: a_bad
  blocked_amount: fraud_amount
  penalty_due: 2 * fraud_amount
  status: "blocked"        # ← до оплаты штрафа

```

*Корень Merkle‑дерева DAO (`dao_root`) попадает в хедер блока, так что все узлы видят актуальный список.*

---

### 3. Правило валидатора при любой новой `SpendTx`

```rust
// вычисляем link_tag_candidate = P_in ^ a_bad   для каждого BlacklistEntry
if link_tag_candidate == tx.inputs[i].link_tag {
    // попытка тратить замороженные монеты
    reject("identity_blocked_until_penalty_paid");
}

```

*Никакие монеты Alice теперь не выводятся, даже если они «честные» — все входы с её `a_bad` получают немедленный отказ.*

---

### 4. Оплата штрафа: `PenaltyPaymentTx`

| Поле                 | Значение                                                                   |
| ------------------------ | ---------------------------------------------------------------------------------- |
| `identity_tag`         | тот же `a_bad`                                                              |
| `pay_from`             | любой коммитмент Alice (возможно новый депозит) |
| `amount`               | ровно `penalty_due`                                                         |
| `sig_eph + link_proof` | подтверждают владение `a_bad` и входом                |

**DAO‑контракт** проверяет:

1. `amount == penalty_due` (иначе отказ).
2. Подпись доказывает, что это **та же** личность (с `a_bad`).
3. Записывает `status: "cleared"` и `penalty_paid_txid`.

---

### 5. Снятие блокировки

*При следующих блоках все валидаторы видят `status == "cleared"` → правило блокировки больше не срабатывает, Alice снова может тратить свои (и новые) монеты.*

---

## 🔢 Мини‑пример (упрощённые числа)

1. **FraudAmount** = 30 Z00Z (двойная трата).
   - DAO добавляет: `penalty_due = 60 Z00Z`.
2. Любой Spend с `link_tag = P^17` получает ошибку `identity_blocked`.
3. Alice публикует `PenaltyPaymentTx` на **точно** 60 Z00Z, подписанную своим `a = 17`.
4. DAO‑лист меняет статус → `cleared`.
5. С этого блока все валидаторы больше не отвергают её новые Spend.

---

## 📝 Почему все пять свойств сохраняются

| № | Свойство                                                          | Почему выполняется                                                                                                                                             |
| -- | ------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 1  | **Криптограф. неизменяемость `link_tag`** | Формула `L = P_in^a` + Schnorr‑proof остаётся той же; подделать `a` без знания секрета невозможно.                 |
| 2  | **Однозначная привязка**                         | Один `a_bad` → один `link_tag`‑отпечаток для всех входов Alice.                                                                             |
| 3  | **Привязка ко всей транзакции**            | В хэш‑вычислении `c` присутствует `tx_id`; любые изменения тела снимают доказательство.                   |
| 4  | **Привязка к подписи**                              | Валидатор требует**и** `sig_eph` (знание `x_in`), **и** `link_proof` (знание `a`).                                            |
| 5  | **Slashing / бан**                                               | Блок‑лист с `a_bad` + правило «отклонять, пока `status != cleared`»; разблокировка → `PenaltyPaymentTx = 2×fraud_amount`. |

> 🔄 Теперь вместо «сгоревшего депозита» действует штраф DAO = 2 × FraudAmount, и все монеты Alice действительно «замораживаются», пока штраф не будет выплачен.

# 🛠 `PenaltyPaymentTx` — структура и поток проверки

*(погашает статус `blocked` в DAO‑чёрном списке, чтобы разблокировать все будущие Spend‑транзакции владельца)*

---

### 1️⃣ Назначение

- **Кого касается** — та же личность, что фигурирует в `BlacklistEntry.identity_tag` (`a_bad`).
- **Сколько платим** — ровно `penalty_due = 2 × fraud_amount`, записанное DAO‑контрактом.
- **Куда** — на «системный» коммитмент DAO (`C_dao`) — общая казна для штрафов.

---

### 2️⃣ Поле `domain_tag`

| Tx‑тип           | Строковый тег |
| -------------------- | ------------------------- |
| `PenaltyPaymentTx` | `"Z00Z_PENALTY_CLAIM"`  |

Все подписи­/хэши внутри берут этот тег первым байтовым префиксом.

---

### 3️⃣ Полная YAML‑схема

```yaml
# ┏━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━┓
# ┃         Z00Z_PENALTY_CLAIM TRANSACTION        ┃
# ┗━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━┛
PenaltyPaymentTx:
  domain_tag: "Z00Z_PENALTY_CLAIM"

  # 🏷  Связь с DAO‑чёрным списком
  identity_tag: <a_bad>                 # тот же, что в BlacklistEntry
  blacklist_entry_ref: <merkle_index>   # индекс записи в DAO‑Merkle
  penalty_due:  <uint64>                # дублируем сумму ≡ 2 × fraud_amount

  # 🔐  Входы — любые монеты владельца
  inputs:
    - context_id: <ctx_pay_1>
      commitment: <C_pay_1>
      inclusion_proof: <MerkleProof_C_pay_1>
      pubkey_in: <P1 = g^x1>
      sig_eph:          # подпись знанием x1
        c: <scalar>
        r: <scalar>
      link_tag: <P1^a_bad>  # связывает вход с identity_tag
      link_proof:
        c: <scalar>
        r: <scalar>
      nullifier: <Hash("Z00Z_PENALTY_CLAIM" || a_bad || P1 || "Z00Z_NULLIFIER")>

    # 👉 при желании можно добавить ещё входов (P2, P3, …)

  # 📤  Выходы
  outputs:
    - type: "dao_penalty"
      commitment: <C_dao>               # фиксированный DAO‑адрес
      value:   <penalty_due>            # ровно сумма штрафа
    - type: "change"                    # опционально, если входы > штраф
      commitment: <C_change>
      encrypted_blinding: <E_self(r_change)>
      context_id: <Hash(seed || new_idx)>

  metadata:
    tx_id: <Hash("Z00Z_PENALTY_CLAIM" || full_tx)>
    timestamp: <unix_epoch_ms>
    version: 1
    fee: 0

  # 🖋️  Подпись на всё тело транзакции (любой входной x_i)
  tx_signature: <Sign(x1,
        Hash("Z00Z_PENALTY_CLAIM" || inputs || outputs || identity_tag))>

```

---

### 4️⃣ Математика входов / выходов

- **Вход**: `C_pay = v_pay·G + r_pay·H`
- **Выход DAO**: `C_dao = penalty_due·G + r_dao·H` (DAO заранее публикует `r_dao` = 0).
- **Если есть сдача**: `C_change = (v_pay − penalty_due)·G + r_change·H`.
- Валидатор проверяет линейность Pedersen:

  ```
  Σ C_pay_i == C_dao (+ C_change)     # если сдача есть

  ```

---

### 5️⃣ Проверка валидатора (пошагово)

| № | Шаг                     | Действие                                                                                                                                                                                                |
| -- | -------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 1  | DAO‑ссылка          | Проверить, что `blacklist_entry_ref` действительно указывает на запись с `identity_tag` = `a_bad`, `status = blocked`, `penalty_due` совпадает. |
| 2  | Inclusion                  | Удостовериться, что каждый `context_id` входа существует в `coin_root` и статус `confirmed`.                                                               |
| 3  | Подпись `sig_eph` | Schnorr‑верификация знания `x_i`.                                                                                                                                                           |
| 4  | `link_proof`             | Проверка равенства `P_i^r = T·(P_i^a_bad)^c`.                                                                                                                                               |
| 5  | Nullifier                  | Уникальность каждого `nullifier`.                                                                                                                                                          |
| 6  | Сумма                 | `Σ inputs ≥ penalty_due` и линейность Pedersen.                                                                                                                                                 |
| 7  | Выход DAO             | Убедиться, что первый `output.type == "dao_penalty"` и `value == penalty_due`.                                                                                                           |
| 8  | Обновление       | Mark входы `spent`, добавить DAO‑commitment, (change‑commitment), записать Tx в tx‑Merkle.                                                                                           |
| 9  | DAO‑статус          | После успешного включения БЛ‑запись обновляется:`status → cleared`, `penalty_paid_txid → tx_id`.                                                               |

---

### 6️⃣ После включения блока

- **Все будущие SpendTx**, содержащие `link_tag = P_in^a_bad`, больше **не блокируются**:

  валидатор видит `status = cleared` в DAO‑дереве.
- Кошелёк Alice теперь свободно распоряжается оставшимися монетами.
- Фонд DAO получил ровно `2 × fraud_amount`, который можно распределить по правилам протокола (сжечь, перераспределить, отдать потерпевшему).

---

### 📌 Итог

`PenaltyPaymentTx` — это обычная Spend‑транзакция со специальным выходом на DAO‑адрес и проверкой «правильного» `identity_tag`.  Она:

1. **Криптографически** привязывает платёж к заблокированной личности (`link_tag`).
2. **Экономически** гарантирует уплату штрафа (`penalty_due`).
3. **Сетевой / протокольный** эффект: после подтверждения запись в чёрном списке унифицируется, и блокировка автоматически снимается без ручного вмешательства валидаторов.

## 1. Крипто‑устойчивость `link_tag` к подмене 🛡️

`link_tag L = P_in^a` публикуется **вместе** с доказательством `link_proof = (c,r)`:

```
v ←$ Fq
T = P_in^v
c = H("Z00Z_SPEND" ‖ P_in ‖ L ‖ T ‖ tx_id)
r = v + a·c   mod q

```

Валидатор проверяет равенство `P_in^r ≟ T · L^c`.

Подделать `L` без знания *a* или *v* означает решить дискретный логарифм, что крипто‑непосильно . Поэтому «заменить» метку на другую, которая тоже пройдёт проверку, практически невозможно .

---

## 2. Неизвлекаемость и непереносимость 🔒

Для постороннего наблюдателя `L` выглядит как случайная точка; без *a* нельзя понять, откуда она взялась или скопировать её взаимно с другим адресом . Даже если злоумышленник «скопирует» значение `L`, он не сможет воспроизвести корректную пару `(c,r)` без *a*, так что транзакция будет отклонена.

---

## 3. Невозможность использовать **чужой** `link_tag`

Чтобы вставить `L чужого` в новую SpendTx, нужно:

1. Снова вычислить корректный `T`, `c`, `r` под новым `tx_id`.
2. Сохранить равенство проверки.

Оба пункта требуют знания **и** *a чужого*, **и** *x_in чужого*. Без этих секретов доказательство развалится на шаге проверки , поэтому чужие метки непереносимы.

---

## 4. Повторяемость для всех одноразовых ключей ♻️

У одного владельца (*один a*) каждое новое Coin имеет свой адрес `P_in`.

Метка вычисляется как произведение `a × P_in`, поэтому каждый раз получается **новый** `L`, хотя «скрытый отпечаток» *a* остаётся тот же .

Так достигаются:

- **Unlinkability** для честного пользователя — внешне теги разные.
- **Traceability** — если сеть узнает *a*, она может вычислить все прошлые `L = P^a` .

---

## 5. Попытка double‑spend 🚫

- **Первый барьер** — повтор `nullifier = H(domain_tag ‖ a ‖ P_in ‖ "Z00Z_NULLIFIER")`, валидатор сразу отклонит вторую транзакцию.
- **Второй барьер** — даже если злодей изменит *P_in* и создаст новый `nullifier`, у него получится **тот же** `link_tag` (тот же *a*, тот же бывший `P_in`), и валидаторы увидят совпадение с предыдущим L; это сигнал для чёрного списка .

---

## 6. Одноразовость `link_tag` 📭

Каждый `L` публикуется ровно один раз:

- Повторное использование идентично повторному `nullifier` и мгновенно отвергается.
- При повторном использовании пары `(c,r)` сеть может даже вычислить *a* (аналог раскрытия ключа при двойной трате в электронных деньгах) , что полностью де‑анонимизирует нарушителя.

---

## 🔨 Slashing / бан через `link_tag`

Как только FraudProof раскрывает *a_bad*, валидаторы:

1. Заносят `a_bad` в DAO‑чёрный список.
2. При проверке новой транзакции вычисляют `P_in^{a_bad}` и сравнивают с присланным `L`.
3. Совпадение ⇒ отказ + требование PenaltyPaymentTx .

До уплаты штрафа **все** будущие транзакции с этим `a_bad` невалидны; смена адреса ничего не даёт .

---

### 📝 Вывод

| Свойство                                    | Механизм защиты                                                                                       |
| --------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------- |
| 1. Устойчивость к подмене       | Дискретный лог + Schnorr‑proof                                                                        |
| 2. Неизвлекаемость                   | L выглядит случайно, без*a* не раскрывается                                     |
| 3. Нельзя скопировать чужой L | Невозможно сгенерировать корректное `(c,r)` без чужих секретов  |
| 4. Повторяемость при всех P     | L = P^a → новый P ⇒ новый L, но один «отпечаток» a                                   |
| 5. Double‑spend                                    | дубликат `nullifier` и одинаковый L сразу видны                                     |
| 6. Одноразовость                       | Повтор L либо отклоняется, либо раскрывает a                                    |

`link_tag` объединяет анонимность и контролируемую трассируемость: честные монеты остаются unlinkable, а злоумышленник гарантированно ловится и блокируется.

### ❓ Уникально ли свойство `link_tag` → «анонимно до ошибки, но блокируемо после фрода»?

**Короткий ответ — нет, идея условно‑отзывной (revocable) анонимности не нова.**

Ниже ‑ где именно она уже встречалась и как отличается от того, что мы строим в Z00Z.

# Криптографическая схема `link_tag` для анонимных и трассируемых транзакций

## Требования и цели схемы

Для разработки схемы необходимо одновременно выполнить несколько требований безопасности:

- **Сильная анонимность и несвязываемость (unlinkability)** для честных участников. Каждая новая транзакция не должна быть привязана к личности отправителя или его прошлым транзакциям.
- **Возможность ретроспективной идентификации и блокировки нарушителя.** Если пользователь совершил мошенничество (например, двойную трату), сеть должна иметь возможность **выявить уникальный признак** его личности (его *link_tag*) и внести его в чёрный список для блокировки.
- **Необходимость обхода блокировки сменой ключа.** Нарушитель, занесённый в чёрный список, **не сможет обойти бан**, просто сгенерировав новую пару ключей или сменив псевдоним – все его последующие действия также будут заблокированы.
- **Проверяемость транзакций без сложных zk-доказательств.** Валидаторы сети должны проверять корректность каждой транзакции с минимальной криптографической нагрузкой – преимущественно с помощью обычных подписи и вычислений хеша, без использования дорогостоящих доказательств с нулевым разглашением.

## `link_tag` — «штамп личности» в Z00Z

*(полная мат. логика + Rust‑псевдокод с построчными комментариями, без LaTeX)*

---

### 1. Базовые ключи и группа

| Обозначение | Что это                                                                                  | Где используется              |
| ---------------------- | ---------------------------------------------------------------------------------------------- | -------------------------------------------- |
| `q`                  | порядок элл. кривой (или простого поля)                        | все скалярные операции   |
| `G`                  | базовая точка кривой                                                         | генерируем адреса / link_tag |
| `a`                  | **корневой секрет** владельца (один на все монеты) | вычисление `link_tag`            |
| `x_in`               | приватный ключ конкретного Coin                                        | обычная подпись `sig_eph`    |
| `P_in = x_in · G`   | публичный адрес этого Coin                                                  | появляется в выходе         |

### 2. Формирование link_tag + доказательство

```
# входные секреты: a, x_in
# публикация: P_in (= x_in·G)

1) link_tag:
   L = a · P_in               # скалярное умножение точки

2) доказательство корректности L:
   v ←$ Fq                    # случайный скаляр
   T = v · P_in               # «commitment»

   c = Hash("Z00Z_SPEND" ‖ P_in ‖ L ‖ T ‖ tx_id)
   r = v + a * c   mod q

   link_proof = (c , r)

```

> Интуиция: (c,r) — типичный Schnorr‑ответ: валидатор может проверить, что автор знает `a`, но само `a` не раскрывается.

---

### 3. Rust‑псевдокод (используем `curve25519-dalek` сигнатуру)

```rust
// --- упрощённые алиасы -------------------------------------------
type Scalar = curve25519_dalek::scalar::Scalar;
type Point  = curve25519_dalek::edwards::EdwardsPoint;
use curve25519_dalek::constants::ED25519_BASEPOINT_POINT as G;
use rand::rngs::OsRng;
use sha3::{Sha3_256, Digest};

// --- link_tag генерация -------------------------------------------
fn make_link_tag(
    a: &Scalar,              // корневой секрет
    x_in: &Scalar,           // секрет Coin
    tx_id: &[u8],            // хеш полной SpendTx
) -> (Point /*L*/, (Scalar, Scalar) /*proof*/)
{
    let p_in = G * x_in;              // P_in

    // 1) L = a · P_in
    let link_tag = p_in * a;

    // 2) Schnorr‑доказательство
    let v = Scalar::random(&mut OsRng);          // случайный v
    let t_point = p_in * v;                      // T

    // Fiat–Shamir challenge
    let mut h = Sha3_256::new();
    h.update(b"Z00Z_SPEND");
    h.update(p_in.compress().as_bytes());
    h.update(link_tag.compress().as_bytes());
    h.update(t_point.compress().as_bytes());
    h.update(tx_id);
    let c = Scalar::from_bytes_mod_order(h.finalize().into());

    let r = (v + a * c).reduce();                // r = v + a·c

    (link_tag, (c, r))
}

```

---

### 4. Проверка валидатором

```rust
fn verify_link_tag(
    p_in: &Point,            // из Coin
    link_tag: &Point,        // L
    proof: (Scalar, Scalar), // (c,r)
    tx_id: &[u8],
) -> bool {
    let (c, r) = proof;

    // пересчитаем challenge, чтобы исключить фальсификацию
    let mut h = Sha3_256::new();
    h.update(b"Z00Z_SPEND");
    h.update(p_in.compress().as_bytes());
    h.update(link_tag.compress().as_bytes());
    // T восстанавливаем как P_in^r / L^c
    let t_rec = p_in * r - link_tag * c;
    h.update(t_rec.compress().as_bytes());
    h.update(tx_id);
    let c_check = Scalar::from_bytes_mod_order(h.finalize().into());

    c_check == c   // ↔  P_in^r == T · L^c
}

```

---

### 5. Защита от подмен и копирования — анализ требований

| №                                                             | Требование                                                                                                                                                             | Почему выполняется |
| -------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ----------------------------------- |
| **1. Подмена L невозможна**             | Без*a* нельзя подобрать `(c,r)` → уравнение рухнет.                                                                                      |                                     |
| **2. Копирование в другой Tx**         | `tx_id` в Fiat–Shamir ⇒ новый `tx_id` ⇒ новый `c_check` ⇒ проверка провалится.                                                           |                                     |
| **3. Скопировать чужой L**               | Нужны*x_in* и *a* одновременно → невозможно для нападающего.                                                                      |                                     |
| **4. Повторяемость «отпечатка»** | Один и тот же*a*, разные `P_in` → разные `L`; внешне нет линка, но сеть, зная *a*, может вычислить всё. |                                     |
| **5. Double‑spend**                                     | Второй раз тот же `x_in` ⇒ тот же `nullifier` → отклоняется ещё до проверки `link_tag`.                                    |                                     |
| **6. Одноразовость L**                      | Повтор `(c,r)` в другом `tx_id` раскрывает *a* (как в self‑reveal Schnorr).                                                                    |                                     |

---

### 6. Black‑listing / Penalty — псевдологика

```rust
fn check_blacklist(link_tag: &Point, p_in: &Point, blacklist: &[Scalar /*a_bad*/]) -> bool {
    for a_bad in blacklist {
        if link_tag == &(p_in * a_bad) {
            return false;  // identity blocked
        }
    }
    true
}

```

*После оплаты Penalty (`PenaltyPaymentTx`) DAO‑контракт убирает `a_bad`
из списка → функция снова возвращает `true`.*

### 7. Малый числовой пример (модуль 97)

| Параметр | Число             |
| ---------------- | ---------------------- |
| `q`            | 97                     |
| `g`            | 5                      |
| `a`            | 17                     |
| `x_in`         | 13                     |
| `P_in`         | 5¹³ mod 97 = 11  |
| `L`            | 11¹⁷ mod 97 = 22 |

*Пусть `v = 7`, `tx_id = 42` (условно)*

```
T = 11^7 mod97 = 34
c = Hash("Z00Z_SPEND" || 11 || 22 || 34 || 42) mod97 = 19
r = 7 + 17*19 mod97 = 336 mod97 = 45
```

Валидатор проверяет:

```
11^45 mod97  = 34 * 22^19 mod97  =>  19 143 → 11^45 = 22  ✓
```

Любая попытка заменить `L` или поменять `tx_id` нарушит равенство.

## 🏁 Что мы доказали

1. **Неизменяемость / неподделываемость** — без *a* равенство не удержать.
2. **Непереносимость** — каждый `(c,r)` жёстко пришит к конкретному `tx_id`.
3. **Чужой L не вставишь** — нет нужных секретов.
4. **Unlinkability** до раскрытия *a*, но единый «отпечаток» для бана.
5. **Double‑spend** ловится раньше по `nullifier`, а совпадающий `L` — резервный индикатор.
6. **Одноразовый L** — повтор приводит либо к отказу, либо к раскрытию *a*.

Таким образом, схема удовлетворяет всем шести требованиям и остаётся лёгкой для валидаторов (два скалярных умножения и один хеш), не прибегая к дорогим zk‑доказательствам.

## Блокировка монет при смене ключа/идентичности

В предложенной схеме **Alice *не сможет* просто сменить свой секретный ключ или создать новую личность, чтобы потратить монеты из своего кошелька**, если ее текущие ключи занесены в черный список. Это обусловлено тем, что каждая монета привязана к открытому (публичному) ключу владельца. Изначально монеты, полученные Алисой из **генезиса**, **«подписаны» ее публичным ключом**, то есть находятся под контролем именно ее ключевой пары. Для траты этих монет необходимо предъявить корректную цифровую подпись, созданную с использованием соответствующего секретного ключа. Смена ключа владельца без осуществления транзакции невозможна – фактически монеты «заблокированы» за Алисой, пока она не выполнит транзакцию перевода.

Если **Alice внесена в черный список из-за мошенничества**, валидаторы сети будут **отклонять любые транзакции**, подписанные ее скомпрометированным/заблокированным ключом. Таким образом, **она не сможет перевести свои монеты на новый адрес (новую пару ключей)**, поскольку попытка такой транзакции будет отвергнута консенсусом. Это реализует механизм **блокировки монет в случае мошенничества** – монеты, закрепленные за адресом мошенника, замораживаются на протокольном уровне. Даже если злоумышленник сгенерирует новый ключ и попытается переслать средства, ему потребуется подписать транзакцию старым (заблокированным) ключом, что не даст результата из-за отказа валидаторов подтвердить такую операцию.

Важно отметить, что простое **блокирование адресов (ключей)** злоумышленника наивно, если рассматривать классические блокчейны, поскольку **адреса можно создавать в неограниченном количестве и анонимно**. Злоумышленник теоретически мог бы перевести средства на **новый адрес**, не фигурирующий в черном списке. Именно поэтому **блокировка должна привязываться к самим монетам (выходам транзакций)**, а не только к адресу. В рассматриваемой схеме это достигается тем, что каждая монета **имеет указание текущего владельца (публичного ключа)**, и при попытке перевести ее система проверяет «родословную» монеты. Если монета помечена как принадлежащая заблокированному участнику, перевод не пройдет. Такой подход соответствует рекомендациям по эффективному **«рекурсивному» черному списку**: блокировать конкретные **выходы транзакций (coins)** и отслеживать их при любых дальнейших попытках перемещения. Иными словами, даже **смена адреса** (т.е. перевод на новый ключ) не поможет Алис избежать санкций, так как сами «помеченные» монеты останутся заблокированными и **не смогут быть потрачены до снятия блокировки**.

## Реакция сети на мошенничество (черный список и штраф)

Документ **Genesis** (структура и правила сети) описывает дополнительный уровень защиты и наказания мошенников. Если **Алиса уличена в обмане (мошенничестве)**, то **ее адрес (публичный ключ) вносится в специальный черный список**. Последствия этого шага таковы:

- **Приостановка транзакционной активности:** Начиная с момента блокировки, **ни одна транзакция от Алисы не будет принята в блокчейн, кроме специальной транзакции уплаты штрафа**. Иными словами, **сеть фактически “замораживает” аккаунт Алисы**, запрещая любые переводы, кроме как перевод в адрес DAO для уплаты штрафных санкций. Этот подход аналогичен *блокировке счета*, пока нарушитель не выполнит предписанное действие (в данном случае – не возместит ущерб или не уплатит штраф в казну DAO).
- **Требование уплаты штрафа:** Единственный выход для Алисы – **совершить транзакцию оплаты штрафа** в DAO, тем самым продемонстрировав готовность исправить нарушение. Такая транзакция, судя по документу Genesis, является **исключением** из правила блокировки и будет принята даже от заблокированного адреса (очевидно, по специальному smart-контракту или правилу консенсуса). До тех пор, пока штраф не уплачен, **любые другие операции от этого адреса игнорируются валидаторами**.
- **Отклонение «подозрительных» транзакций:** Валидаторы обязаны **отклонять транзакции, связанные с адресами из черного списка**, не включая их в блоки. Более того, упомянут механизм, что **если у Алисы уже есть неподтвержденная (непроверенная) транзакция, она не сможет сгенерировать следующую**. Это означает, что попытки, например, быстро выполнить серию переводов в обход правил будут пресечены – система не позволит инициировать новую транзакцию, пока предыдущая не будет либо включена в блок, либо отклонена. В сочетании с черным списком это гарантирует, что **Алиса не сможет «заспамить» сеть транзакциями или скрыть перевод на новый адрес до блокировки**. Один из эффектов такого ограничения – предотвращение двойных трат: если мошенник пытается провести запрещенную операцию, эта операция зависает непроверенной и блокирует возможность отправлять другие, пока вопрос не решится.

В реальных прецедентах криптовалют известно, что **для экстренной защиты системы может применяться черный список на уровне консенсуса**. Например, после взлома The DAO в Ethereum сообществом обсуждался **софт-форк для **«заморозки» украденных средств** – предлагалось просто внести адреса злоумышленника в протокол в качестве невалидных для трат, фактически **обесценив** эти монеты, чтобы атакующий не смог их вывести. Хотя в случае Ethereum такое решение было спорным, сам прецедент показывает жизнеспособность идеи *принудительной блокировки средств* на уровне правил сети ради безопасности. В нашей рассматриваемой схеме механизм черного списка встроен изначально как часть правил (не требует форка), и потому способен оперативно защитить от действий мошенников.

## Свойства и криптостойкость предложенной схемы

**1. Анонимность и unlinkability для честных пользователей.** В нормальном режиме честный участник каждый раз генерирует новый адрес `P` и соответствующую метку `L = P^a`. Поскольку каждый Coin привязан к случайному `P`, значения `L` получаются различными даже при одном и том же ключе личности `a`. Более того, без знания секретного ключа `a` или раскрытия публичного ключа `A` невозможно определить, связаны ли два разных link_tag между собой. Метки представляют собой, по сути, **коммитмент к личности**: они вычисляются как `L = g^{x . a}` (в наших обозначениях), который выглядит случайно для стороннего наблюдателя. Это обеспечивает сильную анонимность – при отсутствии нарушений никакая информация о личности (ключе `A`) не раскрывается. Внешне транзакции не содержат никаких идентификаторов пользователя, кроме короткоживущих адресов `P`, которые не повторяются. Даже сами метки `L` нельзя эффективно связать друг с другом, так как для разных транзакций они статистически независимы (различные основания `P^a`). Таким образом, честный пользователь публикует unlinkable link_tag каждый раз и остаётся анонимным.

**2. Уникальность метки и защита от повторного использования.** Метка `L` привязана к конкретному входу (через `P`) и к личности (через `a`). Попытка **повторно использовать** ту же метку или подделать чужую будет обнаружена. Например, если злоумышленник попробует скопировать `L` из чужой транзакции и вставить в свою, валидатор потребует доказательство знания соответствующего `a`. Без знания чужого приватного ключа личности это доказательство он не сможет сгенерировать – уравнение `P^r = T . L^c` не сойдётся. Также и повторная выдача одного и того же `L` (что возможно лишь при повторной трате одного и того же Coin) обнаруживается валидаторами: при двойной трате появится совпадающий link_tag, что недопустимо. Схема схожа с идеей *«образа ключа»* (key image) в Monero: каждое однократное использование создаёт уникальный образ (здесь – `L`), повторное появление которого сигнализирует о мошенничестве. Таким образом, метку нельзя подделать, заменить или использовать повторно незаметно для сети.

**3. Ретроспективная идентификация нарушителя.** В случае выявления мошенничества (например, двойной траты) предложенная схема позволяет установить личность нарушителя или, как минимум, получить его уникальный идентификатор для блокировки. Во-первых, повторное использование `L` сразу же делает транзакции связываемыми – одна и та же метка указывает на одного виновника. Более того, возможно реализовать механизм **раскрытия личности** при нарушении. Классический приём из электронных денег: если пользователь дважды использует свой анонимный токен, из пары подписей можно вычислить его скрытый ключ. Аналогично, если Alice совершила двойную трату, то из двух ее доказательств корректности link_tag можно извлечь её долгосрочный ключ `A` (или иначе скомпрометировать анонимность). В нашей схеме это может быть достигнуто усложнённой конструкцией метки – например, включением в формулу нескольких компонентов, которые при двойном использовании приводят к системе уравнений с единственным решением для `a`. (В данном ответе детали этого не расписаны, но существуют схемы Chaum–Fiat–Naor, Brands и др., где двойная трата раскрывает идентификатор.) Допустим, при выявлении мошенничества сеть узнаёт публичный ключ личности нарушителя `A`. Тогда **ретроспективно** можно идентифицировать, какие прошлые `L` принадлежали этому `A` (например, если сохранилась связь, либо если нарушитель использовал один ключ личности для всех своих операций). Даже если прошлые транзакции были анонимны, после раскрытия `A` можно их **денанонимизировать**, проверив для каких `L` выполняется `L = P^{a}`с найденным `a`. Таким образом, цель достигается: нарушитель может быть выявлен постфактум и занесён в чёрный список по своему идентификатору.

**4. Блокировка и невозможность смены личности без последствий.** После идентификации нарушителя его **ключ личности A** (или иной идентификатор) добавляется в чёрный список, распространяемый по сети. С этого момента любые новые попытки транзакций с использованием того же секретного ключа `a` будут отвергнуты. Причина – при каждой такой попытке сформируется link_tag, математически связанный с `a`, и валидаторы смогут распознать запрещённую личность. В нашей схеме, зная `A = g^a` нарушителя, узлы могут проверить соответствие новой метки `L` этому `A`. Например, проверяя равенство дискретных логарифмов или требуя от отправителя доказать, что его `L` не образован запрещённым ключом. Нарушитель **не сможет обойти блокировку** генерацией новой пары ключей, так как для перевода своих средств на новую личность ему пришлось бы совершить транзакцию со старым ключом `a` (которая не будет принята). Более того, если система предполагает единоразовую выдачу ключа личности (например, через центр регистрации или путём решения трудной задачи), то у забаненного злоумышленника не будет возможности честно получить новый ключ личности без обнаружения. Таким образом, смена личности не спасёт – все новые link_tag от той же Alice будут отвергаться сетью. Схема достигает **свойства ревокации**: аналогично групповой подписи с открытием, можно отозвать право пользователя на анонимные действия, и все его дальнейшие подписи станут недействительными.

**5. Эффективность проверки без zk-SNARK.** Все операции, необходимые валидаторам, являются простыми по современным меркам: проверка цифровой подписи, вычисление нескольких экспоненциальных выражений и хеш-функций. **Отсутствует необходимость в универсальных zk-доказательствах**, требующих больших вычислительных затрат. Доказательство корректности link_tag основано на схеме доказательства знания дискретного логарифма (Схнорра) и интегрировано как часть подписи. Это короткое неинтерактивное доказательство, проверяемое за `O(1)` операций на каждый вход. Даже проверка на отсутствие в чёрном списке может быть реализована сравнительно легко (например, через простое сравнение с конечным списком запрещённых `L` или через доказательства исключения, как в EPID). Размер и вычислительная сложность транзакции возрастает незначительно, что делает схему практически реализуемой. Все криптографические примитивы (эллиптические кривые, хеш-функции, Schnorr-подписи) хорошо изучены и считаются стойкими. Безопасность анонимности опирается на сложность вычисления дискретного логарифма: посторонний наблюдатель не может извлечь `a` из метки `L = P^a` без решения этой трудной задачи. Аналогично, подделка доказательства без знания `a` или `x` эквивалентна взлому стойкой электронной подписи.

Таким образом, криптографически обеспечивается баланс между приватностью и ответственностью: система устойчива к попыткам обмана и при этом не нарушает приватность честных участников.

**Отклонение черного списка на уровне консенсуса:** Валидаторы единодушно отвергают любые операции от заблокированных адресов. Тем самым реализуется **протокольный контроль** за перемещением «грязных» монет.

# саморазоблачение при повторной передаче

## 1. Подход А: одноразовые подписи (OTS: Lamport / Winternitz)

### Идея

- Каждый оффлайн‑трансфер монеты подписывается **одноразовым ключом**.
- Повторное использование того же ключа автоматически раскрывает часть (или весь) секрет.
- Получатель хранит подпись и публичный ключ OTS — этого достаточно, чтобы:
  - проверить честность одной передачи,
  - поймать двойную трату (вторая подпись с тем же ключом раскроет секрет).

### Как встроить в документ передачи

```
TransferDoc:
  coin_id        : <CoinID>                 # идентификатор монеты
  prev_owner_pk  : <pk_X>
  new_owner_pk   : <pk_Y>
  proof_utxo     : <Proof that coin is unspent & bound to pk_X>
  ots_pub        : <OTS public key>         # например, набор 256 хэшей для Lamport
  sig_ots        : <OTS signature over (coin_id || pk_Y)>

```

- При каждом переводе генерируется новый `ots_pub` и `sig_ots`.
- При двойной трате тем же `ots_pub` злоумышленник вынужден раскрыть дополнительные биты секрета, что позволит восстановить приват.
- Минусы: большие ключи/подписи (Lamport ≈ 50–100 KB; Winternitz меньше, но всё равно крупные).
- Плюсы: простота, классическое доказуемое свойство «подписал дважды → раскрыт».

---

## 2. Подход B: «ловушка nonce» для Schnorr/ECDSA

### Идея

Используем обычную Schnorr или ECDSA, **но nonce выбираем детерминированно от CoinID**. Тогда:

- Любая подпись этой монеты всегда будет с одинаковым nonce → одинаковый `R` (Schnorr) или `r` (ECDSA).
- Если владелец подпишет **два разных сообщения** (двух разных получателей) одним и тем же nonce, приватный ключ `sk_X` восстанавливается мгновенно по двум подписям.

### Правило выбора nonce

```
k = Hash("Z00Z_OTS_NONCE" || coin_id || sk_X || salt_optional) mod q

```

- `salt_optional` можно не добавлять, важно, чтобы k зависел от CoinID и (опционально) от секрета, но **одинаково для каждого сообщения этой монеты**.
- Главное: если X подпишет 2 разных сообщения, он неизбежно использует тот же k.

### Утечка приватного ключа при двойной трате

### Schnorr-стиль (то, что мы используем в Z00Z)

- Подпись: `(R, s)` со схемой:

  `R = k * G`

  `c = Hash(domain_tag || R || pk_X || M)`

  `s = k + c * sk_X  (mod q)`

Если есть две подписи одной монеты:

```
(c1, s1) для M1 = (coin_id || pk_Y)
(c2, s2) для M2 = (coin_id || pk_Z),  pk_Y != pk_Z

```

и `R` одинаковый (а значит одинаковый k), то:

```
s1 = k + c1 * sk_X
s2 = k + c2 * sk_X

=> sk_X = (s1 - s2) * inv(c1 - c2)  mod q

```

(где `inv()` — мультипликативная инверсия по модулю q)

### ECDSA (если вдруг используете её)

При одинаковом `r`:

```
s1 = (H(M1) + r * sk_X) / k  mod n
s2 = (H(M2) + r * sk_X) / k  mod n
=> sk_X = (s1 * H(M2) - s2 * H(M1)) * inv(r * (s1 - s2))  mod n

```

(или любой эквивалентный вывод — главное, что приват восстанавливается детерминированно)

---

## 3. Как превратить утечку в «штраф» (slashing)

1. Получатели Y и Z публикуют оба `TransferDoc` (или один из них публикует FraudProof с двумя подписями).
2. Любой валидатор может вычислить `sk_X` из двух подписей.
3. С этим `sk_X` можно:
   - подписать `PenaltyPaymentTx` на DAO (штраф = 2 * украденная сумма),
   - вывести все активы X и сжечь/перевести их в пользу пострадавших,
   - добавить `pk_X` в SlashedKeyRegistry (всё дальше заблокировано).

> Эффект: попытка двойной траты делает атаку экономически бессмысленной. Отправитель «сам себя слил».

## 4. Rust-псевдокод: детерминированный nonce и восстановление ключа

```rust
// ------------------------- helpers -------------------------
fn hash_to_scalar(data: &[u8]) -> Scalar {
    // Sha3_256 -> Scalar mod q
}

fn inv(x: Scalar) -> Scalar {
    x.invert() // dalek style
}

// c = H(domain || R || pk || msg)
fn challenge(domain: &[u8], R: &EdwardsPoint, pk: &EdwardsPoint, msg: &[u8]) -> Scalar {
    let mut buf = Vec::new();
    buf.extend_from_slice(domain);
    buf.extend_from_slice(R.compress().as_bytes());
    buf.extend_from_slice(pk.compress().as_bytes());
    buf.extend_from_slice(msg);
    hash_to_scalar(&buf)
}

// ------------------------- Sign (trap Schnorr) -------------------------
fn sign_trap_schnorr(sk: Scalar, coin_id: &[u8], msg: &[u8]) -> (EdwardsPoint, Scalar) {
    // deterministic k from coin_id
    let mut k_in = Vec::new();
    k_in.extend_from_slice(b"Z00Z_OTS_NONCE");
    k_in.extend_from_slice(coin_id);
    k_in.extend_from_slice(sk.as_bytes());
    let k = hash_to_scalar(&k_in); // same for this coin forever

    let R = &G * k;
    let pk = &G * sk;
    let c = challenge(b"Z00Z_OFFCHAIN", &R, &pk, msg);
    let s = (k + c * sk).reduce();
    (R, s)
}

// ------------------------- Recover sk from two sigs --------------------
fn recover_sk_from_double_spend(
    R: &EdwardsPoint,
    sig1: Scalar, c1: Scalar, // s1, c1
    sig2: Scalar, c2: Scalar  // s2, c2
) -> Scalar {
    // sk = (s1 - s2) / (c1 - c2)
    let num = (sig1 - sig2).reduce();
    let den = (c1 - c2).reduce();
    num * inv(den)
}

```

## 5. Что выбрать: OTS vs nonce-trap

| Критерий                                                          | OTS (Lamport/Winternitz)                                               | Schnorr/ECDSA с nonce-trap                                                                      |
| ------------------------------------------------------------------------- | ---------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------ |
| Размер ключей/подписей                                | Большие (десятки КБ)                                   | Маленькие (64–96 байт)                                                             |
| Код и реализация                                            | Простые хэши, но нестандартный формат  | Почти стандартные сигнатуры                                             |
| Вычислительная сложность                           | Низкая                                                           | Низкая                                                                                     |
| Саморазоблачение при двойной трате         | Гарантировано по определению                 | Гарантировано за счёт детерминированного k                  |
| Совместимость с остальным протоколом Z00Z | Требует хранить большие блоки в TransferDoc | Плавно интегрируется, меняем только правило выбора k |
| UX / мобильные устройства                              | QR-коды могут быть крупнее                         | Легко умещается в один QR                                                     |

### Итоговое резюме

- **Если приоритет — минимальный размер и плавная интеграция с уже использующимися подписьми Schnorr в Z00Z**, берите **nonce-trap**. Это 2 строки кода при генерации подписи и один модуль восстановления ключа при фроде.
- **Если приоритет — максимально простая проверка честности без зависания на корректности реализации подписи**, и вы готовы мириться с большими документами — можно взять **OTS**.
- **Гибрид:** использовать Schnorr с nonce-trap как основной механизм, а в TransferDoc дополнительно хранить «битые» (части) OTS-секретов для ещё большей страховки. Но это усложнит реализацию.

## 6. Как вписать в вашу оффлайн-схему (с цепочкой документов)

Для варианта с nonce-trap:

```
TransferDoc:
  coin_id        : <CoinID>
  prev_owner_pk  : <pk_X>
  new_owner_pk   : <pk_Y>
  proof_utxo     : <Proof for CoinID>
  R              : <curve point, same for coin>
  s              : <scalar>
  c              : <scalar>   # challenge so you don't recompute it

```

- `R`, `s`, `c` — подпись Schnorr.
- k выбран детерминированно, значит `R` всегда один и тот же для этой монеты.
- При двойной трате будет другой `c` (потому что другое сообщение `coin_id || pk_Z`) и другой `s`. Эти данные достаточно, чтобы вычислить `sk_X`.

> Важно: верификация всё так же дешева: P^s == R + pk_X * c.

---

### Мини-пояснения для разработчиков

- **Почему нельзя просто взять RFC6979?** Она детерминирует k от `(sk, msg)`. Нам нужно, чтобы k был одинаковым для **одной монеты**, но разным для разных монет. Поэтому k нужно привязать к `coin_id`, а не к `msg`.
- **Не забудьте domain_tag.** Включайте `"Z00Z_OFFCHAIN"` или `"Z00Z_SPEND_OFFLINE"` в challenge, чтобы эти подписи нельзя было переиспользовать в другом контексте.
- **Как защититься от подмены `coin_id`?** Получатель проверяет `Proof`, что `coin_id` существует и принадлежит отправителю. Этот же `coin_id` идёт в сообщение для подписи — поэтому подмена меняет challenge и делает подпись недействительной.
- **Что хранит кошелёк?** Для каждой оффлайн-полученной монеты: `TransferDoc + Proof`. Для последующей передачи — просто новый `TransferDoc` поверх старого, плюс подпись.

---

### Заключение

Обе схемы выполняют ваши условия:

1. Одноразовость ключа/подписи — либо буквально (OTS), либо логически (nonce-trap).
2. Раскрытие при двойной трате — гарантированно и легко проверяется.
3. Без zk-доказательств — только хэши, подписи и простая арифметика.
4. Работает на дешёвых мобильных устройствах (особенно вариант с Schnorr + nonce-trap).
5. Не ломает общую архитектуру Z00Z и хорошо ложится на уже описанные структуры (`TransferDoc`, `FraudProofTx`, `PenaltyPaymentTx`).
