### ==Почему Nova / SuperNova==

- Это **IVC-модель** (Incrementally Verifiable Computation), не “ZK-система”: тебе не нужен сокрытый witness, только верификация.
- Рекурсия встроена нативно: каждое новое доказательство содержит проверку предыдущего.
- Нет trusted setup, нет маскирования — чистая верификация состояния.
- Уже реализована на Rust, хорошо встраивается в собственные протоколы.
- Proof компактный (сотни байт-килобайт), проверка быстрая.

### Почему не чистый STARK

- Да, он прозрачный и без trusted setup, но рекурсия там громоздка и требует отдельного уровня агрегации.
- STARK-доказательства большие (мегабайты) → плохо для “root-в-блокчейне”.

### Почему Plonky2 — «план Б»

- Он рекурсивный и очень быстрый (использует FRI и field-friendly циклы).
- Если тебе потом нужно “доказать” что-то в zk-режиме — можно включить.
- Минус: довольно большой кодовый стек, изначально ориентирован на zk-rollup-ы.

### 👉 Резюме

| Что взять                     | Почему                                                       |
| ----------------------------- | ------------------------------------------------------------ |
| 🥇 **Nova / SuperNova (Rust)** | Прямая IVC, без ZK, чисто рекурсивная цепочка доказательств состояния. |
| 🥈 **Plonky2 (Rust)**          | Быстрая, ZK-готовая альтернатива, если потом нужно zk-агрегация. |
| 🥉 **STARK (Winterfell)**      | Только если хочешь полностью прозрачную схему и большие proof’ы тебя не пугают. |

### Размер доказательства

- **Nova**: изначально спроектирована как IVC — каждое следующее доказательство «пришивает» предыдущее, но сами пруфы остаются довольно компактными (десятки КБ, иногда меньше, зависит от кривой и схемы). Её цель — как раз *удерживать размер вменяемым при росте числа шагов*.
- **Nova** нормально живёт на CPU, потому что шаг рекурсии у неё “тонкий” — она для этого и писалась.

## Что у тебя есть

- Есть **дерево монет/nullifier’ов** — пусть это Merkle / sparse Merkle / Verkle, неважно.
- Есть **root_old** — корень дерева до блока.
- Есть набор операций блока: “добавить монету”, “потратить монету → пометить nullifier”, “создать новую”.
- После применения операций у тебя получается **root_new**.

Тебе нужно *доказать*, что `root_new` получен **корректно и строго** из `root_old` по этим операциям.

## Что делает схема на Nova/SuperNova

Внутри proof’а ты описываешь один и тот же переход:

1. **Публичные входы** схемы:
   - `root_old`
   - `root_new`
2. **Приватные входы** (свидетели):
   - список апдейтов (твои монеты/хэши, которые меняются в дереве),
   - меркле-пути для каждого апдейта (чтобы доказать, что именно этот лист был в `root_old` *или* что он там отсутствовал — для вставки),
   - всё, что нужно для построения нового корня.
3. Схема делает:
   - для каждого апдейта: проверяет, что меркле-путь действительно выводится из `root_old`;
   - применяет обновление (insert/update/delete);
   - пересчитывает корень;
   - в конце проверяет: полученный корень == `root_new`.
4. **Рекурсия**: схема ещё и проверяет прошлый proof:
   - из прошлого proof она получает “выходной корень прошлого блока”;
   - утверждает: “он должен быть равен моему `root_old`”.

То есть условие внутри схемы буквально такое:

```text
assert prev_proof.output_root == root_old
assert apply_updates(root_old, updates) == root_new
```

Если это выполняется — генерируется новый proof. И он уже относится к новому корню.

## Что это тебе даёт

- Ты можешь хранить **только корни** по блокам + сами proof’ы.
- Любой, у кого есть последний proof и последний корень, может убедиться, что **вся цепочка обновлений дерева была валидной**.
- При этом сами сырьиные транзакции/монеты можно вынести в оффчейн/архив — они не нужны для ончейн-валидации.

## Важная тонкость

Единственное, что **нельзя выбрасывать**, — это сама “множество запрещённых идентификаторов” (nullifier set), если оно нужно для защиты от двойной траты. Но и его ты можешь:

- либо держать как отдельный большой коммитмент (и доказывать только переход корня этого множества),
- либо шардировать и по шард-корням тоже строить такие же рекурсивные proof’ы.

Так что да: *“держу монеты и хэши в дереве → доказываю Nova/SuperNova, что новое дерево — честная функция от старого”* — это абсолютно нормальная, даже образцовая модель для твоего типа блокчейна.

    Это идеальный вариант: первый блок/снапшот/код перехода лежит открыто, его можно прогнать локально и увидеть, что первый proof действительно про это состояние.
    → Значит, “доверие к генезису” у тебя не слепое, а *проверяемое*.
2. **Доверие к консенсусу (Celestia)**
    Ты не изобретаешь свой выбор цепочки, а говоришь: “правильная последовательность блоков определяется внешним консенсусом”.
    Тогда всё просто: если Celestia сказала “вот этот заголовок следующий” — значит этот proof и этот root канон.
    → Здесь доверие не к автору proof, а к тому, что **правильный заголовок выбран консенсусом**.
3. **Доверие к доступности данных = на стороне отправителя**
    Ты честно говоришь: сеть не обязуется вечно хранить твои сырые tx.
    Ты получил подписанную квитанцию — храни сам.
    Нужно будет доказать что-то через год — предъяви свой чек.
    → Это нормальная ответственность клиента. Это как с наличными: потерял — это твоя проблема.
4. **Что остаётся у сети**
   - корни состояний,
   - рекурсивные доказательства,
   - (возможно) коммитмент на nullifier-set.
      Всё это сеть может хранить и проверять без доступа к приватным данным пользователей.
5. **Permissionless/trust-minimized вывод**
   - Любой может принести новый блок + proof → консенсус решит, включать или нет.
   - Никто не обязан верить “на слово”, потому что proof проверяемый.
   - Единственные места доверия ты озвучил и сделал явными.

То есть да: в таком виде твоя ZUS — *permissionless по входу* и *trust-minimized по вычислению*. Остальное — честно вынесено на пользователя и на выбранный консенсус. Это хорошая, “честная” архитектура.
