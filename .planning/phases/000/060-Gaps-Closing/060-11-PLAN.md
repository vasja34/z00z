<action>
Implement `C4` exactly. Add a verification-pipeline performance inventory that
records wall time, CPU total, CPU percent, max RSS, filesystem output, cache or
target roots, and execution mode for the slowest gates. Normalize reusable Rust
compile work around one stable release feature set where tool soundness allows
it, split compile-heavy work from execution-heavy work, and share workspace
dependency metadata across supply-chain subchecks where they consume the same
inventory. Add explicit resource budgets for `l4-unsafe`, `l4-fuzz`, and
`l4-constant-time` so wall-clock wins cannot hide CPU or RSS regressions. Keep
tool-specific isolation where a tool requires it. After optimization, rerun the
final Phase 060 packet: docs, supply-chain, adversarial review, and the broad
workspace release gate plus the targeted Phase 060 slices they cover. Publish a
short closure memo that records fully closed items, any intentionally deferred
items, and any accepted risks that remain. Preserve the profiling baseline
explicitly for `l3-verify-fast`, `test:workspace`, `l4-supply-chain`, and
`supply:semver:origin/main` so before/after comparisons stay tied to the
numbers reported in the current TODO evidence set. If `060-12`, `060-13`,
`060-14`, or `060-15` lands after the first optimized-tree rerun, refresh the
same final packet and closure memo on that newer tree instead of leaving the
earlier rerun as the carried authority.
