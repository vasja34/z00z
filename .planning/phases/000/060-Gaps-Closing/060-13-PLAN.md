commit.

<threat_model>
- A prepared tx path without explicit negative balance tests invites future
  regressions where the assembler still compiles but stops rejecting malformed
  value or commitment combinations.
- A voucher path without explicit conservation mismatch tests invites silent
  residual drift, accidental cash projection, or reject-code skew between
  storage and RPC surfaces.
- A typed object path that treats `FeeEnvelope` as storage-only validation
  would let preview/build/validator surfaces overclaim acceptance and would
  fork fee-support truth away from the rest of the object package contract.
- Collapsing the shipped native cash fee lane into `FeeEnvelope` inside Phase
  060 would create concept drift against the live corpus and risk replacing
  Pedersen-balanced fee outputs with a different contract surface before a
  dedicated design phase authorizes it.
- Misstating `FeeEnvelope` as an arithmetic or Pedersen-summed fee asset would
  conflate two different invariants and could hide a regression where the
  native fee-output lane weakens while the support-contract lane still passes.
- Forcing rights into asset-like value checks would create concept drift by
  turning authority objects into pseudo-cash just to satisfy a symmetric test
  shape.
- Letting this reopen land after `060-11` without refreshing the targeted
  wallet/object closeout evidence would overstate final Wave 4 attestation on an
  older tree.
</threat_model>
