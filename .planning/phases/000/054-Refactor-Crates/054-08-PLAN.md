# 054-08 Plan: Runtime Digest Rebinding And Planner Canonical Path

<objective>
Close `AS-20260609-001` by binding runtime planner routing to payload-verified
digests and removing bypass seams that still trust caller-supplied digest
metadata.

Purpose: make runtime-owned planner authority fail closed at ingress instead of
borrowing correctness from upstream callers.
Output: one canonical tx/claim intake path whose normalized digest drives
routing, intake ids, and plan digests, plus regression evidence and refreshed
attack-surface records.
</objective>

<coverage_contract>
This plan owns the accepted runtime-planner trust gap recorded in
`AS-20260609-001`. It does not reopen the closed backend seam, placement,
rename, or docs waves except where a doc or validation artifact must be
corrected to describe the new canonical intake contract. The slice must close
the runtime boundary itself: route selection, planner digests, and intake
metadata may no longer rely on caller-supplied `tx_digest_hex` strings or on
optional upstream verifier discipline. Any direct-construction bypass in live
public DTOs or re-exported helper paths must either disappear or become
obviously internal and uncallable from the canonical public path. The digest
formula source is already live in `z00z_wallets`: execution must reuse
`build_tx_package_digest` and `build_claim_tx_digest` exactly, and must not
promote `verify_full_tx_package`, `ClaimTxVerifierImpl::verify`, or rollup-node
theorem checks into a second planner-admission authority. Because the live
public crate also exposes `AggregatorIngress`, `AggregatorOrdering`,
`OrderingBoundary`, and `SchedulerBoundary` on raw `WorkItem`, the narrowing
must cover that whole lane, not only `BatchPlanner`. `IntakeId` may remain a
public output identifier because watcher alerts consume it, but callers must
not retain authority to mint trusted routing digests through that type.
</coverage_contract>

<action>
Introduce one runtime-owned canonicalization step that recomputes the
payload-bound digest for both `TxPackage` and `ClaimTxPackage`, rejects
mismatches before planning, and stores the verified result in a type that
`BatchPlanner` uses for route lookup, ordered intake ids, and `plan_digest`
construction. Reuse the exact live wallet digest builders
`build_tx_package_digest` and `build_claim_tx_digest`; do not fork the digest
formula locally and do not call `verify_full_tx_package` or
`ClaimTxVerifierImpl::verify` as a planner-admission substitute. Narrow the
planner entry path itself, not only helper constructors: the public
`AggregatorIngress`, `AggregatorOrdering`, `OrderingBoundary`,
`SchedulerBoundary`, and `BatchPlanner::{plan_batch,make_batch}` path must
accept only a verified intake type produced by `IngressBoundary::normalize`,
while raw `WorkItem` construction becomes internal or otherwise non-bypassable
from the canonical public path. Preserve `IntakeId` as an output-side runtime
identifier where downstream watchers need it, but remove caller authority to
manufacture trusted route metadata through public struct literals or raw
planner-entry slices. Any public `&[WorkItem]` path that can still carry
caller-filled digest metadata into route lookup or planner scheduling is
forbidden.
</action>
<acceptance_criteria>
- Runtime ingress recomputes canonical digests from payload bytes for both tx and claim inputs and rejects mismatches fail closed.
- `BatchPlanner`, `OrderingBoundary`, `SchedulerBoundary`, and the public aggregator service traits route or schedule only from verified planner metadata, not raw caller-supplied digest strings.
- The canonical public runtime path cannot construct a planner-ready `WorkItem` that bypasses digest rebinding unnoticed, because the public planner and ordering APIs accept only normalized verified intake values and raw DTO construction is no longer a usable bypass.
- Output-side `IntakeId` consumers such as watcher alerts remain supported without reopening input-side digest authority.
- No new parallel verifier or duplicate digest formula is introduced; planner ingress reuses the exact existing wallet digest helpers.
</acceptance_criteria>
