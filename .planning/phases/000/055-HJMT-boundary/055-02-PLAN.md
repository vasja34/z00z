<action>
Implement `BatchProofBlobV1` verification as one storage-owned atomic pipeline:
decode with limits, reject unsupported generation or shard context, reject
table-bound violations, enforce canonical path order, reject duplicate paths,
reject mixed proof families, reject opening-kind mismatches, reject witness
domain or reference misuse, bind the exact encoded bytes into the transcript,
reconstruct the root for every path, and return one accepted or rejected batch
verdict only. Extend the existing proof error surface with typed fail-closed
classes where needed, but do not expose a partial-acceptance API or per-path
success list from a failed batch.
</action>
<acceptance_criteria>
- The verifier enforces canonical ordering, duplicate rejection, mixed-family
  rejection, opening-family rejection, reference-bound checks, and root binding
  before reporting success.
- A failed batch never returns a partially accepted path set.
- Error reporting stays typed and fail-closed while remaining compatible with
  the storage-owned proof-checking surface.
</acceptance_criteria>
<verify>
1. Run `./.github/skills/smart-tests-bootstrap/scripts/bootstrap_tests.sh` first as the mandatory fail-fast gate. If it fails, stop, fix the issues, and rerun it before any broader validation.

Also run `cargo test -p z00z_storage --release --features test-params-fast --test test_hjmt_batch_proof -- --nocapture` and `cargo test -p z00z_storage --release --features test-params-fast --test test_hjmt_batch_proof_negative -- --nocapture` to prove the verifier surface directly.
3. Then run `/.github/prompts/gsd-review-tasks-execution.prompt.md` through `/GSD-Review-Tasks-Execution` in YOLO mode at least 3 times and continue until at least 2 consecutive runs show no significant issues in the code.
4. When a commit is needed, use `/z00z-git-versioning`.
</verify>

<threat_model>
- Tampering: mutated header, table, or witness bytes can bypass weak decoders;
  the verifier checks limits, structure, and semantics before success.
- Repudiation: a failed batch can appear partially useful; the API never
  exposes partial acceptance.
- Denial of service: oversized tables or invalid indexes can force expensive
  work; prechecks and bounds enforcement happen first.
</threat_model>
