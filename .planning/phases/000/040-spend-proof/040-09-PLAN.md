# 040-09 Honest Continuation Blueprint

## Scope Note

This document continues the reopened Phase 040 work after the 2026-04-27 audit
retraction of `040-08-PLAN.md` and `040-08-SUMMARY.md`.

Its scope is intentionally split into two classes of gaps:

1. **Remaining in-phase gaps** that Phase 040 still can and should close.
2. **Residual out-of-phase gaps** that must be recorded truthfully but must not
   be disguised as already-closed Phase 040 work.

This plan therefore does **not** assume that the current approved Tari-backed
surface is already sufficient for the full spend theorem. It also does **not**
reinterpret `040-12 Checkpoint-Pipeline Reuse` as a standalone
checkpoint/state-transition proof architecture.

It also does **not** select or land later proof-family candidates that belong
to later product or protocol phases rather than this reopened spend-proof
boundary: encrypted-transfer validity/equality or fee proofs, portable
non-membership accumulator proofs, blind issuance primitives, or signature-stack
swaps such as Schnorrkel.

## Objective

Close the remaining honest Phase 040 spend-proof gaps without reintroducing
fabricated closure language.

The continuation target is:

1. make one repository-backed decision about the real spend-theorem backend on
   the approved seam,
2. either upgrade the live suite truthfully or keep it reopened truthfully,
3. close the live public-verifier negative-matrix debt,
4. synchronize all Phase 040 ledgers to the same live boundary, and
5. hand off the residual checkpoint/state-transition and rollup proof gaps as
   explicit future work instead of hiding them under Phase 040 wording.

## Pros/Cons Decision Gate

Every material decision in this plan must be recorded with explicit pros and
cons in execution notes, review notes, and summary artifacts.

### Option A: Force immediate theorem closure on the current Tari Bulletproofs+ surface

Pros:

- keeps one proof family,
- preserves current dependency direction,
- avoids introducing a second proof lane during Phase 040.

Cons:

- the local `z00z_crypto` surface is repository-backed for Bulletproofs+
  **range proofs**, not yet for the full spend-theorem relation set,
- creates a high fabrication-risk path if the backend ends up replaying witness
  rules again instead of verifying a real theorem-carrying proof,
- collapses the capability question into an assumption.

Verdict:

- **Disallowed** unless a repository-backed capability proof is produced first.

### Option B: Capability-first theorem closure on the approved seam

Pros:

- keeps the current approved crypto stack if a real theorem adapter is proven
  feasible,
- avoids a second proof system unless the repository evidence demands it,
- allows the suite transition to be grounded in real backend behavior.

Cons:

- may still fail if the local approved surface cannot express the theorem,
- may end with an honest partial close instead of a full theorem close.

Verdict:

- **Default starting path** for this plan.

### Option C: Keep Phase 040 honest and hand the theorem gap forward

Pros:

- preserves truthful repository state,
- removes false-close language immediately,
- still allows test-matrix and closeout-ledger work to complete.

Cons:

- leaves the spend theorem gap explicitly open,
- requires a later proof-system phase for a real theorem backend.

Verdict:

- **Mandatory fallback** if Option B cannot be proven on the approved surface.
