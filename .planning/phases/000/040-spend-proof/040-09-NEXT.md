## 🧭 Recommended Next Plan Scope

Yes. A concrete next plan should be written.

That next plan should be limited to the following closure path:

1. Define the exact public/private witness contract for a real spend-theorem
   backend on the approved seam.
2. Decide whether the theorem backend can live on the current approved stack or
   whether the seam itself must change.
3. Implement authoritative membership composition against `prev_root` so the
   theorem path is not silently dependent on an external checkpoint-only
   assumption.
4. Add backend, public-verifier, and simulator tests that prove theorem-artifact
   acceptance and legacy rejection without witness replay.
5. Only after that, open the next proof phase for standalone checkpoint or
   state-transition proving, and only after that, open rollup settlement proof
   work.

Things that should **not** be in the first next-plan slice:

- migrating from Tari Bulletproofs+ to the `bulletproofs` crate just to swap
  range-proof libraries,
- accumulator-based non-membership proofs,
- ciphertext equality or fee proofs,
- blind issuance,
- `schnorrkel`-driven signature refactors.

The audited repository does not show that those families are the blocking item
for current `040-09` closure. The blocking item is theorem relation proving on
the live seam.
