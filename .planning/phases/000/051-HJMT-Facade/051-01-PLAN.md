# 051-01 Plan: Storage Facade And Compatibility Backend

<objective>
Introduce the Phase 051 storage backend boundary and move the current shared
namespaced JMT implementation behind it as the compatibility backend.
</objective>

<execution_context>
Read `.planning/phases/051-HJMT-Facade/051-CONTEXT.md`,
`.planning/phases/051-HJMT-Facade/051-TODO.md`, and
`docs/Z00Z-JMT-Design.md` sections 3.1, 3.3, 4.2, 7.2, 9.1, and 12 before
editing.
</execution_context>

<context>
The live `AssetStore` already exposes semantic operations, but its physical
state and public facade are the same struct. This plan separates the facade
from the current implementation without changing the caller-facing storage
contract.
</context>

<action>
Define one storage-owned `AssetTreeBackend` or equivalent facade boundary. The
boundary must cover semantic root reads, `CheckRoot` reads, item lookup, asset
lookup, list pagination, batch `StoreOp` mutation, compatibility proof queries,
claim-source proof reads, checkpoint-facing reads, and reload validation hooks.
Keep `AssetPath`, `StoreItem`, `StoreOp`, `AssetStateRoot`, `CheckRoot`,
`ProofItem`, `ProofBlob`, and `ProofScanOut` as the semantic contract types.
Do not expose `TreeId`, namespace bytes, physical branch order, or raw JMT keys
through this boundary. Reuse the existing `AssetModel`, `TreeStore`,
`RedbBackend`, `ProofBlob`, checkpoint modules, and path-index machinery. Do
not create a duplicate semantic model, duplicate proof verifier, duplicate
checkpoint verifier, duplicate path-index authority, or dummy forest backend.
Document that future forest commit journal and crash-recovery behavior must
enter behind this same boundary, while compatibility mode delegates to the
current RedB and checkpoint validation paths.
</action>
<acceptance_criteria>
- A single named backend trait or facade boundary exists in `z00z_storage::assets`.
- The boundary has methods or associated helper types for root, check-root,
  lookup, batch mutation, proof, claim-source, checkpoint-facing reads, and
  reload validation.
- No public method on the boundary accepts `TreeId`, namespace bytes, branch
  ordering, raw physical keys, or raw backend-root authority inputs.
- Existing public `AssetStore` method names remain available to downstream
  callers.
- Source comments or docs state that forest commit journal and crash-safe
  publication are future backend capabilities behind the facade, not a Phase
  051 compatibility-mode reimplementation.
- No new module duplicates the semantic model, proof verifier, checkpoint
  verifier, RedB reload validation, or path-index authority.
</acceptance_criteria>

<action>
Move the current shared namespaced JMT behavior behind a `CompatibilityBackend`
or equivalent backend implementation. Keep `AssetStore` as the stable public
facade and delegate its existing semantic methods to the compatibility backend.
Preserve current RedB load, rehydrate, path index, claim nullifier, version
history, snapshot, and checkpoint artifact behavior. Keep rollback on failed
commits equivalent to the current `snap_store()` and `restore_store()` behavior.
Do not copy the current implementation into a parallel layer; move or wrap the
existing code in place so one compatibility path remains authoritative.
</action>
<acceptance_criteria>
- The current shared namespaced JMT implementation has an explicit compatibility
  backend name in source.
- `AssetStore::new`, `AssetStore::load`, `put_item`, `del_item`, `apply_ops`,
  `proof_item`, `proof_blob`, `proof_scan`, `root`, `check_root`, and
  `claim_source_contract_for_item` preserve existing behavior.
- Existing store API tests still verify public roundtrip, path contract,
  namespace separation, root contract, duplicate path rejection, delete
  pruning, no-op root, and reorder-stable root behavior.
- The compatibility backend is described in source comments as the reference
  backend, not the target forest performance architecture.
- Tests or source-shape checks show that there is one compatibility backend
  path, not both old direct implementation and new duplicated implementation.
</acceptance_criteria>

</threat_model>
