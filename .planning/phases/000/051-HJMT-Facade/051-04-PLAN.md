<action>
Create a golden compatibility matrix that runs semantic operations through the
new facade and compares results against the compatibility backend reference.
Cover insert-many, delete-many, hot-serial, cross-definition, duplicate path,
delete-missing, reorder-stable roots, no-op roots, asset lookup, path lookup,
list pagination, replay behavior, and proof verification outcomes. Structure
the harness so a future forest backend can be added as another backend case
without changing the golden expectations. Do not add a dummy forest backend or
copy compatibility logic into a fake second backend merely to satisfy the
matrix; until a real forest backend exists, the harness should name the future
backend slot and run the compatibility backend as the only executable case.
</action>
<acceptance_criteria>
- Golden tests assert identical `AssetStateRoot` outcomes for insert-many,
  delete-many, hot-serial, and cross-definition workloads.
- Golden tests assert duplicate path and delete-missing reject without changing
  root or stored items.
- Golden tests assert proof verification succeeds for valid compatibility
  witnesses and fails for wrong semantic inputs.
- The harness has a named compatibility backend case and a clear future backend
  extension point, but Phase 051 does not ship a production forest backend.
- The harness fails closed or skips explicitly labeled future-backend cases
  rather than comparing compatibility against copied compatibility logic.
</acceptance_criteria>
