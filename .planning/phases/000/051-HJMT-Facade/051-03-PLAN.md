# 051-03 Plan: API Guardrails And Downstream Cutover

<objective>
Prevent downstream coupling to physical backend details and route validator,
wallet, and simulator consumers through storage-owned semantic authority.
</objective>

<execution_context>
Read `.planning/phases/051-HJMT-Facade/051-CONTEXT.md`,
`docs/Z00Z-JMT-Design.md` sections 4.2, 7.2, 7.3, 9.3, and 12, then inspect
the listed wallet and simulator proof consumers before editing.
</execution_context>

<context>
The code already keeps `TreeId` crate-private, but public proof summaries still
carry diagnostic backend-root bytes. This plan makes diagnostic use explicit
and prevents that value from becoming authority in wallets, simulators, or
tests.
</context>

<action>
Replace any downstream authority decision that depends on raw `backend_root`,
raw `TreeId`, namespace prefixes, branch ordering, or physical key layout with
semantic facade calls, `ProofBlob` verification, `ProofScanOut` diagnostic
checks, `AssetStateRoot`, or `CheckRoot`. Validator checkpoint flow must consume
storage-owned checkpoint authority and the existing verdict or reject vocabulary
instead of copying checkpoint proof formulas or creating a second artifact
schema. It is acceptable for audit records to store backend-root bytes only
when they also store semantic root and root-bind data and do not use
backend-root equality as the authority decision. Reuse existing wallet witness,
claim verifier, spend verifier, simulator storage view, `ScanChunk`, and
`ScanStatePayload` modules; do not add a second validator checkpoint verifier,
wallet storage authority, simulator-owned proof rule, scan cursor model, replay
registry, or copied branch-proof verifier.
</action>
<acceptance_criteria>
- Validator code consumes storage-owned checkpoint authority and does not define
  a second checkpoint proof formula, artifact schema, or verdict vocabulary.
- Wallet and simulator code obtain membership through storage-owned proof APIs
  rather than reconstructing backend branch semantics locally.
- Any backend-root equality check in wallet or simulator code is diagnostic and
  paired with semantic root plus root-bind verification.
- Source-shape tests flag new imports of `TreeId`, `ns_key`, namespace tags, or
  physical branch layout outside `z00z_storage`.
- Existing wallet and simulator proof-verification tests continue to pass.
- No new validator, wallet, or simulator module becomes the authority for
  storage proof verification; storage-owned checkpoint contracts, `chk_blob`,
  `ProofBlob`, `ProofItem`, or facade calls remain the authority.
</acceptance_criteria>
