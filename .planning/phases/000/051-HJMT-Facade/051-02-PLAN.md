# 051-02 Plan: Root Taxonomy And Proof Envelope

<objective>
Codify root taxonomy and make the compatibility proof envelope an explicit
fail-closed version boundary.
</objective>

and `ProofBlob::backend_root()`. This plan turns that separation into an
enforced taxonomy and explicit envelope contract.

<action>
Make root semantics explicit in code and tests. Keep `AssetStateRoot` as the
public asset-state root. Keep `CheckRoot` as the checkpoint-facing typed root.
Keep `TxDigest::to_check()` rejecting with `RootErr::TxRootMix`. Ensure
`SettlementStateRoot` is not exported or referenced as a live runtime type.
If a proof-local backend root wrapper is introduced, keep it private to
storage proof code and prohibit conversion into `AssetStateRoot` or
`CheckRoot`. In `z00z_core`, add only source-shape guardrails if needed: do not
move storage root authority into core, do not export `SettlementStateRoot`, and
do not introduce live `RightLeaf` or `FeeEnvelope` types as part of Phase 051.
</action>
<acceptance_criteria>
- Tests assert that `AssetStateRoot` converts to `CheckRoot` only through the
  intended typed path.
- Tests assert that transaction digests cannot become checkpoint roots.
- Source-shape tests assert there is no public `SettlementStateRoot` export in
  `z00z_storage::assets`.
- Source-shape tests or review checks assert there is no live
  `SettlementStateRoot`, `RightLeaf`, or `FeeEnvelope` runtime export added to
  `z00z_core::assets` by Phase 051.
- Any backend root wrapper remains crate-private or proof-local and has no
  conversion into `AssetStateRoot` or `CheckRoot`.
</acceptance_criteria>

<action>
Define the compatibility proof-envelope version boundary around current
`ProofBlob` behavior. Preserve storage-owned encode/decode. Enforce the
semantic root, path, definition root leaf, serial root leaf, terminal leaf,
leaf hash, backend root binding, and three branch proof segments. Add explicit
reject classes or tests for unsupported version, malformed bytes, wrong
semantic root, wrong path, wrong checkpoint root context, unexpected
compatibility bucket-policy metadata, wrong definition proof, wrong serial
proof, wrong asset proof, and detached payloads. Do not fake deletion or
non-existence proof support in compatibility mode; if those proof families are
not implemented, add explicit unsupported-proof-family rejects and document
that the real deletion or non-existence proof semantics belong to the future
forest backend behind this verifier boundary.
</action>
<acceptance_criteria>
- `ProofBlob` or its wrapper exposes one current compatibility envelope version.
- `chk_blob` or its storage-owned verifier rejects unsupported versions and
  malformed bytes fail-closed.
- Tests cover wrong semantic root, wrong path, wrong parent-root leaves, wrong
  terminal leaf, wrong terminal leaf hash, wrong backend-root binding, and
  wrong branch proof bytes.
- Compatibility mode rejects unexpected bucket-policy metadata instead of
  silently ignoring it.
- Checkpoint-facing proof checks can pass an expected `CheckRoot` context
  without accepting a raw backend root as checkpoint authority.
- Deletion and non-existence proof envelope forms are either implemented with
  explicit verification semantics or rejected with explicit unsupported-family
  errors; compatibility mode must not accept placeholder proofs.
</acceptance_criteria>
