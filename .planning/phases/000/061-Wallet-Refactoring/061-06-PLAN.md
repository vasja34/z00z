# 061-06 Plan: RPC Method Implementation Flattening And Helper Renames

<objective>
Flatten the RPC methods implementation tree into one-level `src/rpc/method_*`
and `src/rpc/test_*` files after the RPC facade has already been stabilized.
</objective>

<coverage_contract>

`D-02`, `D-03`, `D-06`, `D-11`, and `D-14`. It must not reopen the already
stabilized support/logging/types facade except for import wiring required by
the move.
</coverage_contract>

<action>
Move all RPC method implementations into one-level `src/rpc/method_*.rs` and
`src/rpc/test_*.rs` files, including the nested asset, key, storage, backup,
wallet, network, object, tx, and chain methods. Apply the `_rpc_` helper
renames from the TODO so names such as `asset_rpc_*` and `tx_rpc_*` become
`method_asset_*` and `method_tx_*`, with `method_tx_support.rs` used for the
shared tx helper that would otherwise collide with `method_tx_impl.rs`. Update
every `include!`, `mod`, and registration entrypoint so the method surface
stays coherent under the preserved `adapters::rpc` facade.
</action>
<acceptance_criteria>
- No RPC method implementation remains in a nested subtree below `src/rpc/`.
- Redundant `_rpc_` helper names are removed where `src/rpc` now gives enough
  context.
- Method registration entrypoints and method tests still compile and run
  through the preserved facade.
- `asset_impl`, `key_impl`, and `tx_impl` include chains are updated in the
  same slice as the file moves.
</acceptance_criteria>
<verify>
1. Run `./.github/skills/smart-tests-bootstrap/scripts/bootstrap_tests.sh` first as the mandatory fail-fast gate. If it fails, stop, fix the issues, and rerun it before any broader validation.

Run the slice-specific checks: `cargo check --release -p z00z_wallets --all-targets --all-features`, `rg -n "asset_rpc_|tx_rpc_|methods/" crates/z00z_wallets/src/rpc crates/z00z_wallets/src/adapters crates/z00z_wallets/tests`, and `rg -n "register_all_wallet_rpc_methods|register_all_app_rpc_methods" crates/z00z_wallets/src/rpc crates/z00z_wallets/src/adapters`. Then run `cargo test --release` when this slice changes Rust or test files.
3. Run `./.github/prompts/gsd-review-tasks-execution.prompt.md` (`/GSD-Review-Tasks-Execution`) in YOLO mode at least 3 times and fix all issues and warnings. Stop only after at least 2 consecutive runs show no significant issues in the code.
4. If this slice needs a git commit after validation passes, use `/z00z-git-versioning` instead of ad hoc git commit commands.
</verify>

  look like pure file moves.
- Method helper renames can leave split naming vocabularies behind if imports
  are only partially updated.
- Reopening support/logging/types work here would blur the signal when method
  regressions appear.
</threat_model>

<verification>
