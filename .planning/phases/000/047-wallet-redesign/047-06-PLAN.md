## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| build selection -> reservation state | A mismatch could double-spend pending assets or hide reserved ones. |
| tx history sidecar -> owned-asset mutations | Two persistence planes can drift if failure ordering is not explicit. |
| asset RPC views -> compatibility semantics | User-facing responses could keep old placeholder logic while live authority changed. |

| Threat ID | Category | Component | Disposition | Mitigation Plan |

| T-047-17 | R | broadcast/cancel/reconcile | mitigate | Keep submitted/admitted-status writes behaviorally compatible while using fail-closed rollback or one atomic path when evidence/status writes disagree. |
| T-047-18 | I | asset views and compatibility ops | mitigate | Derive balance/details/list from owned-asset status and tx bindings only, keep TOFU/payment-request checks on the build path, and prevent `wallet.asset.*` write-style helpers from becoming a parallel authority. |
</threat_model>
