# Phase 019.1: 019.1-rename - Context

**Gathered:** 2026-03-26
**Status:** Ready for planning

<domain>
## Phase Boundary

Phase 019.1 is a naming-only rename phase. It translates the existing consolidated rename audit in `RENAME-PLAN.md` into crate-scoped GSD plan wrappers, without changing runtime behavior, public contracts, or the vendor boundary.

The phase is split by crate so downstream execution can be owned, reviewed, and summarized per crate: `z00z_core`, `z00z_crypto`, `z00z_simulator`, `z00z_storage`, `z00z_utils`, and `z00z_wallets`.

This phase does not introduce new features, new APIs, or new product scope. It only applies the rename recommendations that already exist in `RENAME-PLAN.md`.
</domain>

<decisions>
## Implementation Decisions

### Source of truth
- **D-01:** `RENAME-PLAN.md` remains unchanged and is the canonical source of rename recommendations.
- **D-02:** The crate plan files are wrappers around the corresponding crate sections in `RENAME-PLAN.md`; they do not reinterpret or redesign the recommendations.

### Crate partitioning
- **D-03:** Create one GSD plan file per crate: `z00z_core`, `z00z_crypto`, `z00z_simulator`, `z00z_storage`, `z00z_utils`, and `z00z_wallets`.
- **D-04:** Each plan file is scoped to a single crate and its direct rename fallout, including re-exports, module path updates, test file renames, and `test_` prefix fixes.

### Safety and scope
- **D-05:** Do not modify `crates/z00z_crypto/tari/`.
- **D-06:** Keep the phase English-only in all technical artifacts.
- **D-07:** Preserve behavior; rename-only changes must not alter semantics, artifact meaning, or test intent.

### Execution order and verification
- **D-08:** Execute the crate plans in codebase dependency order: `z00z_crypto` and `z00z_utils` first, then `z00z_core`, then `z00z_storage`, then `z00z_wallets`, and finally `z00z_simulator`.
- **D-09:** Wrapper file numbering does not override the execution graph; the executable plans' `depends_on` and `wave` fields are the source of truth for sequencing.
- **D-10:** Every auto task must verify in this order: run `./.github/skills/smart-tests-bootstrap/scripts/bootstrap_tests.sh` first, rerun it after every fix until it passes, then run the phase-local release gate, then run the broader `cargo test --release --features test-fast --features wallet_debug_dump` gate when relevant.
- **D-11:** Every auto task must run `/.github/prompts/gsd-review-tasks-execution.prompt.md` (`/GSD-Review-Tasks-Execution`) in YOLO mode at least three times and continue until two consecutive runs report no significant issues.
- **D-12:** If a broader gate fails, the executor must classify the failure as rename fallout or a pre-existing blocker before closing the task.
</decisions>

<specifics>
## Specific Ideas

- The user explicitly wants each crate handled by a separate `PLAN.md` wrapper.
- The user explicitly wants the recommendations in `RENAME-PLAN.md` applied as-is.
- The user explicitly wants the wrapper layer, not a rewrite of the rename recommendations.
- The consolidated rename audit remains useful as the canonical reference document for all crate wrappers.
- The executable plans must keep rename fallout isolated per crate and only widen to cross-crate verification after the local pass is green.
</specifics>
