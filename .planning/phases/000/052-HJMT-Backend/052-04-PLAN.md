<action>
Implement the target forest proof encoding inside storage ownership. The proof
must bind `AssetStateRoot`, `AssetPath`, committed bucket policy fields,
definition root leaf, serial root leaf, bucket root leaf, the terminal leaf,
and all required branch-proof bytes. Verification must recompute the bucket id
from the committed policy, verify the definition, serial, bucket, and terminal
segments against the expected chained roots, and reject malformed envelopes,
wrong path, wrong definition leaf, wrong serial leaf, wrong policy, wrong
bucket, wrong branch, wrong terminal leaf, wrong terminal leaf hash, wrong
semantic root, wrong root binding, wrong backend-root diagnostic binding, or
unsupported forest versions. Record representative serialized inclusion-proof
size for single-leaf and shared-parent multi-leaf workloads as implementation
evidence.
</action>

  `z00z_storage`.

  wrong terminal leaf hash, wrong terminal proof, wrong branch proof, wrong

- Compatibility proofs remain compatibility-owned and are not silently treated

  state unchanged.

<action>
Implement deletion and non-existence proof families only where the live
verifier can validate them fail-closed. Deletion proofs must bind prior root,
deleted path, next root, and the affected parent-root updates. Non-existence
proofs must use canonical default commitments, not node-local `not found`
responses: `DEFAULT_VALUE_COMMITMENT` for terminal absence and
`DEFAULT_CHILD_COMMITMENT` for internal absence. Add replay-consistency checks
for repeated absence proofs at the same root, reject present-key absence
claims, and measure prove time, verify time, serialized proof size, and
aggregation throughput for representative absent-key workloads. If any proof
family cannot be validated honestly, keep it explicitly unsupported. Keep
deletion and non-existence unsupported in compatibility mode unless storage
explicitly lands compatibility-family semantics, and do not introduce
tombstones unless the design source changes first.
</action>
<acceptance_criteria>
- Deletion proofs either validate prior-root to next-root transitions with
  affected parent-root updates or remain explicitly unsupported.
- Non-existence proofs validate canonical default commitments under the exact
  root and path or reject.
- Present-key absence, tampered defaults, tampered path indexes, tampered
  bucket policy, wrong roots, and replay drift reject fail-closed.
- The proof test matrix covers empty-tree absence, random absent keys after
  inserts, replay consistency at the same root, and mixed inclusion plus
  absence under the same parent root.
- Rejecting deletion or absence proof workloads leave semantic store state
  unchanged.
- Compatibility mode keeps deletion and non-existence unsupported unless an
  explicit storage-owned semantic upgrade lands later.
</acceptance_criteria>
