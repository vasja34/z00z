# 052-02 Plan: Forest Tree Store And Batch Planner

<objective>
Build the private physical forest tree store and the deterministic batch
planner that turns `StoreOp` sequences into real bucketed forest commits.
</objective>

recreation of the old shared-commit behavior under a new name. This plan is
the first real implementation slice: it adds the private forest layout and
turns insert or delete workloads into deterministic child-tree work rather
than a copied compatibility path.

when it does not yet exist in the current codebase. If `052-01` or a local
execution step lands a better-fitting extension point first, extend that
landed module instead of creating a duplicate logic lane.

<action>
Add the private tree identities and storage helpers needed for the real forest
layout. The implementation should introduce bucket-level and bucket-local asset
tree ownership behind `store_internal`, keep deterministic sort order for
definition, serial, bucket, and asset updates, and keep physical key
derivation private. Reuse the current store internals where possible, but do
not reuse the compatibility shared flat root as the forest authority path.
Keep path-index writes operational and private instead of letting them become a
second semantic-root authority plane.
</action>
<acceptance_criteria>
- Storage internals can address definition, serial, bucket, and terminal asset
  tree state distinctly without exporting those identities publicly.
- Child asset roots, bucket roots, serial roots, and definition roots have
  deterministic ordering and non-colliding namespace behavior.
- Public assets modules still do not export physical tree identities or
  namespace helpers.
- The forest store can host child JMT state independently of the compatibility
  shared physical root.
- Path-index writes remain separate from the public semantic root and never
  become exported root authority.
</acceptance_criteria>

<action>
Implement the planner that validates every `StoreOp`, groups puts or deletes by
definition, serial, and derived bucket, rejects duplicates or missing deletes
before mutation, and applies deterministic child updates before parent-root
updates. Preserve no-op and idempotent put behavior from the compatibility
corpus, and make the exported `AssetStateRoot` semantically equal to the
compatibility backend for the same operation stream.
</action>
<acceptance_criteria>
- Insert-many, delete-many, hot-serial, and cross-definition workloads run
  through real forest planning and mutation paths.
- Duplicate path and delete-missing workloads reject with state preserved.
- Reordering semantically identical workloads produces stable semantic roots.
- Empty operation sets preserve the current semantic root.
- The planner does not copy compatibility mutation code into a fake second
  backend.
</acceptance_criteria>

  reject before mutation and prove state preservation.
- Public layout leakage could turn physical buckets into caller-managed state;
  keep all tree identities crate-private.
</threat_model>
