<action>
Add `VoucherLeaf` as a committed conditional-claim leaf with object id or
terminal id, issuer commitment, holder commitment, beneficiary commitment,
backing commitment or reserve reference, face value, remaining value, policy
id, action pool id, lifecycle state, validity window, acceptance/refund
metadata, replay nonce, and disclosure/audit commitments. Extend
`SettlementLeaf` with `Voucher(VoucherLeaf)`, add human-readable serde labels,
binary family tags, `From` conversions, `family_tag`, `terminal_id`,
`serial_id`, `as_voucher`, and `check_path` support. Add
`SettlementLeafFamily::Voucher`, marker leaves, proof binding,
family-specific nonexistence, batch proof tags, HJMT proof envelope support,
and HJMT cache encode/decode. Keep existing terminal/right tags stable and add
tests proving old terminal/right rows still decode.
</action>
<acceptance_criteria>
- `SettlementLeaf` includes Terminal, Right, and Voucher families with explicit
  serde and binary tags.
- `SettlementLeafFamily` includes Voucher and can produce marker leaves.
- `SettlementPath` remains one shape for all object families.
- Terminal, Right, and Voucher inclusion, deletion, and nonexistence proofs are
  family-specific.
- Asset paths cannot validate voucher or right leaves; voucher paths cannot
  validate asset or right leaves; right paths cannot validate asset or voucher
  leaves.
- HJMT cache decodes old terminal/right family codes and rejects unknown codes.
- Docs list voucher leaf support and explicitly keep one settlement root.
</acceptance_criteria>

<threat_model>
- Reusing terminal/right family tags for vouchers can corrupt durable proofs.
- A family-agnostic nonexistence proof can let one object family prove absence
  for another.
- A parallel voucher tree would violate the one settlement-root contract and
  split semantic authority.
</threat_model>

<verification>
This plan passes only when all leaf-family boundaries know about Voucher and
old Terminal/Right proof and cache compatibility remains stable.
</verification>
