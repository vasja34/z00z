# 059-02 Plan: Core Object Vocabulary, Policy Descriptors, And Action Pools

<objective>
Create the core vocabulary for object families, voucher config, policy
descriptors, action pools, action ids, condition descriptors, and witness-facing
object roles without changing genesis or storage roots yet.
</objective>

<coverage_contract>
This plan owns object semantics decisions D-05, D-06, D-07, D-08, D-09,
Voucher/Right/FeeEnvelope separation decisions D-44, D-45, D-46, D-47, D-48,
and policy/action decisions D-49, D-50, D-51, D-52, D-53.

Decision coverage:
- D-05: Asset means final spendable value and remains the only cash balance.
- D-06: Voucher means conditional value claim and is not final cash until a validator-checked redeem action creates cash.
- D-07: Right means authority without value and contributes zero to value conservation.
- D-08: CashPolicy for native assets is fixed and narrow, with no arbitrary ActionPool semantics.
- D-09: VoucherPolicy and ActionPool are target architecture terms until implemented.
- D-44: VoucherLeaf carries conditional-claim committed state.
- D-45: Voucher value fields must support redeemed amount, residual claim, and backing/reserve accounting.
- D-46: Voucher holder and beneficiary are separate and transfer cannot silently change beneficiary or refund authority.
- D-47: RightLeaf remains zero-value and rejects value-like fields.
- D-48: Asset, Voucher, Right, and FeeEnvelope remain separate object roles.
- D-49: Policies are content-addressed descriptors with stable canonical bytes and hashes.
- D-50: Native cash policy stays special while voucher/right policies may use deterministic validator-readable action pools.
- D-51: MVP conditions are deterministic or verifier-safe attested; subjective/oracle-heavy conditions are deferred.
- D-52: Policy descriptors declare object families, rights, signatures, lifecycle, expiry, replay, and conservation contribution.
- D-53: Unknown policies fail closed for validators and quarantine in wallets.
</coverage_contract>

<action>
Add core-only types that describe object families and policy/action descriptors
without claiming storage implementation. Define `ObjectFamily` with `Asset`,
`Voucher`, and `Right`. Define stable target ids for `PolicyId`, `ActionId`,
`ActionPoolId`, and condition descriptors if no equivalent type exists. Add
`PolicyDescriptorV1` with canonical bytes/hash rules, allowed input/output
object families, action ids, condition descriptors, required rights, required
signatures, attestations, expiry/replay rules, and conservation contribution.
Add `ActionPoolDescriptorV1` with deterministic action ordering and per-action
input/output/lifecycle/witness requirements. Add `VoucherConfigEntry` as a
core config type with issuer, holder, beneficiary, backing/reserve reference,
face value, remaining value, policy id, action pool id, lifecycle, validity,
acceptance/refund metadata, replay nonce, and disclosure/audit commitments.
Extend rights config validation so any budget, fee, payer, sponsor, support,
reserve, amount, nominal, backing, or value-like key is rejected. Keep native
asset policy fixed and reject arbitrary action pools on native cash.

Populate the currently empty `actions`, `policies`, `rights`, and `vauchers`
module roots as canonical homes or compatibility facades instead of creating a
parallel asset-layer vocabulary. `actions` owns action ids, action pools, and
action descriptors; `policies` owns policy ids, policy descriptors, and
condition descriptors; `rights` owns right-specific authority policy/action
types; `vauchers` owns voucher config, voucher policy, and voucher lifecycle
types. If a symbol remains under `assets` for compatibility, expose it through a
thin re-export to one canonical implementation and add tests that prove the
canonical bytes/hash do not change through either public path.
</action>

<threat_model>
- If voucher policy is added by extending native cash policy, cash becomes
  arbitrary programmable value and violates D-08/D-50.
- If rights accept value-like fields, rights can become hidden vouchers or fee
  budgets.
- If policy descriptors lack canonical bytes and hashes, validators, wallets,
  storage witnesses, and simulator fixtures can bind different semantics.
</threat_model>

<verification>
This plan passes only when core object vocabulary is deterministic, target/live
distinctions are documented in tests, native cash remains fixed-policy, vouchers
carry conditional-claim fields, and rights remain zero-value.
</verification>
