# 059-03 Plan: Core Genesis Policies, Vouchers, And Publication

<objective>
Extend `z00z_core::genesis` under the existing shared orchestration boundary so
assets, rights, policies, and bootstrap vouchers are generated deterministically
and exported through one settlement manifest.
</objective>

<coverage_contract>
This plan owns genesis decisions D-10, D-11, D-12, D-13, D-14, D-15,
compatibility decision D-42, and publication details D-54, D-55, D-56, D-57,
and D-58.

Decision coverage:
- D-10: `z00z_core::genesis` remains the single shared genesis orchestration boundary.
- D-11: Per-object generators live inside that boundary: genesis_assets, genesis_rights, genesis_vouchers, and policy/action-pool generation.
- D-12: Assets keep finite-supply genesis semantics based on class, serials, nominal value, and policy flags.
- D-13: Rights use authority-instance genesis semantics and never mint or reserve value.
- D-14: Vouchers use conditional-claim genesis semantics and every genesis voucher binds backing/reserve evidence or genesis reserve.
- D-15: Policy generation is deterministic and content-addressed.
- D-42: Existing config compatibility is decided before adding required vouchers or policies.
- D-54: Genesis exports object-class artifacts and one settlement manifest.
- D-55: Genesis derivation is domain-separated by network, chain id, object class, object id, index, root generation, and policy/action descriptor hash.
- D-56: Assets, rights, vouchers, and policies use separate typed config sections under one genesis boundary.
- D-57: Genesis vouchers are bootstrap exceptions; runtime issuance remains the ordinary path.
- D-58: Genesis and simulator fixtures include positive and negative object examples for assets, vouchers, and rights.
</coverage_contract>

<action>
Extend `GenesisConfig` with additive `policies` and `vouchers` sections. Keep
existing asset finite-supply semantics and existing right authority-instance
semantics. Decide and implement compatibility for missing or empty rights,
policies, and vouchers so existing devnet/testnet/mainnet/simulator fixtures do
not fail only because new arrays are absent. Add `genesis_policies` to generate
canonical policy/action descriptor records and exported hashes. Add
`genesis_vouchers` to generate explicit bootstrap voucher records only when
configured, requiring backing/reserve evidence, issuer, holder, beneficiary,
face value, remaining value, lifecycle, policy id, action pool id, validity,
refund/acceptance metadata, replay nonce, and audit commitments. Domain-separate
all derivations by network, chain id, object class, object id, index, root
generation, and policy/action descriptor hash. Extend settlement manifest export
with policy and voucher artifacts while preserving `genesis_rights.json` and
`genesis_settlement_manifest.json` semantics. Bind `genesis_policies`,
`genesis_vouchers`, and `genesis_rights` to the canonical `policies`,
`vauchers`, `rights`, and `actions` modules from `059-02`; do not duplicate
descriptor structs, canonical hash code, lifecycle enums, or validation logic in
the genesis module.
</action>

<threat_model>
- New required config arrays can break all existing genesis and simulator
  fixtures if compatibility is not planned.
- Reusing right derivation labels for vouchers can create object identity and
  audit ambiguity.
- Bootstrap vouchers without reserve/backing evidence can mint conditional
  claims with no conservation anchor.
- Private genesis-local policy or voucher structs can drift from validator,
  wallet, storage, and simulator semantics even when tests compile.
</threat_model>
